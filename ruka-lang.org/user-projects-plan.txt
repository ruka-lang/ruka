User Projects — Playground Implementation Plan
===============================================

A persistent, multi-file project workspace inside the playground. Projects
live in IndexedDB, separate from the bundled examples. Only `.ruka` and
`.txt` files are supported. `.txt` files are treated as inert notes — no
highlighting, no parsing, no execution.


Phase 1 — Persistence layer  [DONE]
------------------------------------
- Project schema with explicit folder list and per-file kind (`ruka` | `txt`).
- IndexedDB store keyed by id; record carries name, schemaVersion,
  files, folders, entry, createdAt, updatedAt.
- API: listProjects (sorted by updatedAt desc), loadProject, saveProject,
  deleteProject, renameProject, newStoredProject, toProject.
- Helpers: fileKindFromPath, foldersFromFiles, projectFromExample,
  updateFileSource (immutable).
- Tests: 7 unit tests covering kind classification, folder derivation
  (nested/root/dedup), example→project conversion, immutable updates.

Files: src/lib/playground/project.ts, storage.ts, project.test.ts


Phase 2 — Project lifecycle UI  [DONE]
---------------------------------------
- Active source: discriminated union { kind: "example" | "project", id }.
- Single grouped <select> (Examples / My Projects) via SelectGroup support.
- Buttons: NEW (always), SAVE AS (examples only), RENAME / DELETE
  (user projects only), RUN.
- Empty template = single `main.ruka` with a `let main = () do … end`
  scaffold. Top-level scope checker rejects bare statements, so the
  template must be a proper declaration.
- Autosave: 500ms debounced, fires only when active.kind === "project".
- `.txt` files skip type check.

Files: src/routes/playground/+page.svelte, src/lib/components/ui/select.svelte


Phase 3 — File tree sidebar + file CRUD  [DONE]
------------------------------------------------
- Project mutation helpers (addFile, addFolder, renameFile,
  renameFolder, deleteFile, deleteFolder, setEntry, pathExists) with
  unit tests covering entry rewrites, cascade delete, ancestor folder
  registration, and folder-rename descendant rewriting.
- FileTree component: hierarchical tree from (files, folders),
  collapsible folders, hover-revealed action buttons.
- Per-file actions: open, set-as-entry (.ruka only), rename, delete.
  Delete disabled for the entry file.
- Per-folder actions: new file, new folder, rename, delete (cascades;
  blocked when the entry lives inside the folder).
- New file picker restricted to `.ruka` / `.txt`; popover shows an
  inline error when the name fails validation or collides.
- Workspace layout: 220px sidebar | editor + terminal column. Stacks
  below 720px. FileTabs deleted.

Files: src/lib/playground/project.ts, project.test.ts,
src/lib/components/file-tree/index.svelte,
src/routes/playground/+page.svelte


Phase 4 — `ruka.import` builtin  [DONE]
----------------------------------------
- `ruka.import("util.ruka")` is a special form recognised in the scope
  checker, type checker, and evaluator. Path is project-relative against
  the importing module's directory; bare names work, `./` and `../`
  segments resolve as expected, absolute paths and root escapes are
  rejected.
- Privacy by case: only lowercase top-level bindings end up in the
  imported module's exported member set (`ModuleType.members` /
  `ModuleValue.members`).
- Cycle detection in both type checker and evaluator. Diagnostics carry
  a `path` field; the playground prefixes cross-module errors with
  `in {path}: …`.
- New `src/lib/interpreter/project.ts`: ProjectContext, path resolver,
  parsed-AST cache, `checkProject(sources, entry)` orchestrator.
- `checkScope` and `checkTypes` accept optional `(ctx, path)` and behave
  unchanged in single-file mode. `run` accepts optional
  `(RuntimeProject, entry)` and caches `ModuleValue` per path.
- Driver: `checkProjectSource` and `runProject`. Playground calls them
  whenever a project has an entry, so imports are live as you type.
