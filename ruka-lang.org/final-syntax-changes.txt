Final syntax/semantics/docs changes before beginning compiler implementation
- [ ] Docs & [ ] Interpreter: Tuple type annotations written as `(Type, Type, ...)`
	The only difference between this and a function annotations is the missing `->` which makes sense to me as a argument list is essentially a tuple and a parameter list is essentially a tuple destructure semantically
- [ ] Docs & [ ] Interpreter: Privacy handled with `local` keyword for bindings as `local name = ...` or fields `record { local x: type, ... }`
- [ ] Docs & [ ] Interpreter: local runtime bindings can't be captured by closures
- [ ] Docs & [ ] Interpreter: Multi line literals ie. records, arrays, tuples don't require commas
- [ ] Docs & [ ] Interpreter: `.` removed from literals:
	Arrays become `[Type] { e, e, ... }` or `{ e, e, ... }` for type inference
	Records become `Type { f = v, f = v, ... }` or `{ f = v, f = v, ... }` for type inference
	Variants become `some(...)` and `none`
		- For inference, the compiler would check if these match identifiers first, if not then check members of in scope variants
	Tuples become `(Type, Type, ...) (e, e, ...)` or `(e, e, ...)` (no option for type prefixing which is not neccessary)
- [ ] Docs & [ ] Interpreter: Add builtin map type
	Type annotations follow `[Key => Value]` and literals `[Key => Value] { key => value, ... }` or `{ key => value, ... }`
- [ ] Docs: A compile-time type parameter can be inferred by where the result is stored when final arg is a named param with a type of type
- [ ] Docs: Static receivers (Lets call them type receivers and self receivers) are typically used for attaching statics to foreign types. A file can represent a single type following the pattern
	```
		// Vector.ruka
		let t = ...
		let new = (...) -> t do ...
		let method (self) = () do ...

		// main.ruka	
		local Vector = ruka.import("Vector.ruka")

		let main = () do
			let v: Vector.t = Vector.new(...)
			v.method()
		end
	```
	Applying directly to t is unneccessary and would make the code more cumbersome. But still allowing type receivers may be useful for extending types particularly from the stdlib or third-party libraries
- [ ] Docs & [ ] Interpreter: Type inference does not synthesize types. All values must match a predefined type. Type inferences figures out what type annotations would have be written based on the types in scope with any ambiguity being a compile error
- [ ] Docs: Integers and floats can be implicitly cast to larger types otherwise must cast explicitly
- [ ] Docs: Casting keyword `value as type` that uses a ruka.cast behaviour
- [ ] Docs: Methods also support move mode & and maybe stack and compile-time mode (which would require the receiver to be in a corresponding binding) I'm just the use cases for those yet
- [ ] Docs: Closures (runtime functions)
- [ ] Docs: Code creating or inspecting type values can only be run at compile-time
- [ ] Docs: More destructuring functionality
	- [ ] `if pattern = var do ...` executes block if pattern matches
	- [ ] `for pattern in iter do ...` skips elements that don't match
	- [ ] `while pattern = var do ...` terminates on first element that doesn't match
- [ ] Docs: Fields = runtime record memory, Members = constants attached to a type. Refer to statics as members.
- [ ] Docs & [ ] Interpreter: Records with no fields cannot be instanciated.
- [ ] Docs: Remove any reference of modules from docs. Files are record types that have no fields, only members. Refer to them as files or records as appropriate.
- [ ] Docs: Array comprehensions are missing from the reference
- [ ] Docs: Packages called Bouquets `ruka bouquet new` and index files are traditionally called petals `petal.ruka`
- [ ] Docs: Library only packages use `src/petal.ruka` as their entry
- [ ] Docs: /docs should contain the grammar and reference information as well so it's in a simple form readable without all the html
	These would be the single source of truth used by /ruka-lang.org and /ruka
