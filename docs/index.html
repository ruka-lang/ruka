Core Features

    
    
      
  • 
      	Compiled and/or Interpreted. The ruka compiler uses the interpreter for metaprogramming (allowing generics and complex constants, etc.), and users can use the interpreter for rapid debugging, extending the build process, and scripting.
      
    • 
      
  • 
      	Value-based. Functions, types, records (structs), variants (tagged unions), and tests are all values stored in bindings. Files/modules are imported as records so are values as well.
      
    • 
      
  • 
        Expression-based. Blocks, conditionals, match expressions, and function bodies all evaluate to a value. The last expression in a block is its result.
      
    • 
      
  • 
        Garbage collected with opt-in lifetime control. The GC handles memory by default but parameter/binding modes allow the user to decide when to control lifetimes or stack/heap allocate.
      
    • 
      
  • 
        Structural typing. Shared functionality is expressed through behaviours: any type matching the method signatures satisfies an behavour implicitly and can be passed to Behaviour typed parameters. Parameters typed with a behaviour cause that function to be compiled for the argument type passed in.
    
      
    • 
      
  • 
        Built-in testing. Test functions are first-class language constructs declared with test name = .... They compile only in debug and test builds — zero overhead and zero code in production binaries.

Development


    
Philosophy

    
    
      
  • Fully functional interpreter. The language should be runnable without compilation, enabling rapid iteration and scripting use cases.
    • 
      
  • Custom compiler backend. Ruka targets its own backend rather than relying on LLVM or similar, keeping the toolchain self-contained and tightly integrated with the language.
    • 
      
  • Self-hosted tooling. The compiler, interpreter, and tools are eventually written in Ruka itself.
    • 
      
  • A singular tool. Project management, package management, testing, and editor integration are all provided under one unified ruka command — no fragmented ecosystem.
    •

Future Plans

      

        The current focus is on the core language. Functionality allowing custom compiler/interpreter extensions to be made for domain specific solutions is being considered with the following examples also being considered.
      


      

        
Ruka.web

        

          First-class JSX-like syntax for HTML and CSS directly in Ruka source, enabling server-side
          rendering without a template engine. Components would be ordinary Ruka functions.
        

      


      

        
Ruka.hdl

        

          A hardware description language using Ruka syntax, targeting FPGA development on
          SoC boards such as the Xilinx Zynq family. Hardware modules expose getter and setter
          behaviours, allowing the same language to describe both the software and the hardware it runs on.
