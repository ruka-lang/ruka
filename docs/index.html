Ruka

A statically typed, garbage-collected, expression-based programming language.

What is Ruka?

Ruka is a general-purpose compiled programming language designed around clarity and expressiveness. It is statically typed with pervasive type inference, garbage collected, and expression-based — nearly every construct produces a value.

Ruka has compile-time metaprogramming allowing static dispatch, generics, and generating constants that require more complex calculations; dynamic dispatch through interfaces and opaque types; mutable semantics and opt-in lifetime control.

Declarations in Ruka are uniform: functions, types, and modules are all anonymous values stored in named bindings. There is no special declaration syntax — only const, local, and let.

Design Principles

  • Uniform declarations. Functions, types, and modules are all values stored in bindings. There is one syntax for naming things.
  • Expression-based. Blocks, conditionals, match expressions, and function bodies all evaluate to a value. The last expression in a block is its result.
  • Garbage collected with opt-in lifetime control. The GC handles memory by default. When precise lifetime management is needed, parameter modes give the programmer explicit control without abandoning the GC model entirely.
  • Compile-time as a first-class feature. Types, functions, and modules are first-class values in compile-time contexts. The eva parameter mode marks values that must be known at compile time, enabling generic programming via static dispatch.
  • Interfaces over inheritance. Ruka has no inheritance. Shared behaviour is expressed through interfaces, giving the function an opaque representation of the argument only seeing a subset of it's behaviour.
  • Methods live outside types. A type definition describes only memory layout. Methods and static functions are declared separately in the same scope, keeping type definitions minimal and readable. Co-location is also what enables type inference: the compiler knows every type in scope and can resolve unannotated receivers and record literals from field access alone.

Core Features

Strong Static Typing

Aggressive type inference — annotations are only written where the type cannot be uniquely determined. Ambiguity is a compile error, never a silent fallback.

Garbage Collection

Memory is managed automatically. Parameter modes allow opt-in lifetime control.

Expression-Based

Blocks, if/else, and match all return values. No statement/expression split.

Pattern Matching

Exhaustive match over variants, literals, and guard conditions.

Records & Variants

Product types (records) and sum types (tagged union variants).

Interfaces

Structural typing via interfaces. Any type matching the method signatures satisfies an interface implicitly. Parameters typed with an interface use dynamic dispatch automatically.

Compile-Time Evaluation

Ruka code can be interpreted at compile time to generate types, instantiate generics, and generate constants that require more complex calculations.

First-Class Types

Types and modules are values that can be passed to compile-time (eva) parameters.

Built-in Collections

Arrays, tuples, and hashmaps are built into the type system with literal syntax.

A Quick Look

The example below defines a Dog record type, a public static constructor, a public method, a private mutable method, and a main entry point. Note that methods are declared outside the type definition. 

// Type definition — describes memory layout only
local Dog = record {
    name:  string,
    breed: string,
    age:   isize
}

// Public static function — receiver type in parentheses before `=`
const new (Dog) = (name, breed) do
    Dog.{ name, breed, age = 0 }
end

// Public method — named receiver with type annotation
const about (self) = () do
    "My name's ${self.name}. I'm a ${self.breed}."
end

// Private mutable method — `mut` mode before the receiver identifier
local age_up (mut self) = () do
    self.age = self.age + 1
end

const main = () do
    local ruka = Dog.new("Ruka", "Jack Russell Terrier")

    // Pipeline: passes left-hand value as first argument to the next function
    ruka.about()
        |> @println()
end

Parameter Modes

By default all values are passed as immutable references managed by the garbage collector. A mode keyword placed before a parameter identifier changes how the compiler treats the lifetime and ownership of that value.

ModeBehaviourUse case
mut Mutable reference. Changes are visible to the caller. In-place mutation of a value or collection.
mov Transfers ownership into the function. The caller can no longer use the value. Resource transfer, unique ownership patterns.
stc Stack-allocated copy. Cannot escape the function scope. Small, short-lived values where GC overhead is undesirable.
eva Value must be known at compile time. Evaluated by the compiler interpreter. Generic programming, compile-time type generation. 
const push (self: List) = (mut item: i32) do
    // item is a mutable reference
end

const consume = (mov resource: Resource) do
    // resource ownership transferred here
end

const process = (stc buf: Buffer) do
    // buf is a stack-allocated copy; cannot escape this scope
end

// Using parameter `t` as a type infers `t: type` and implies eva
// generic function, instantiated per concrete type
const min = (t, a: t, b: t) => if a < b do a else b end

const main = () do
    local x = min(i32, 3, 7)     // compiler instantiates min for i32
    local y = min(f64, 1.5, 2.0) // compiler instantiates min for f64
end

Future Plans

The current focus is on the core language. The following extensions are planned for the future.

Ruka.web

First-class JSX-like syntax for HTML and CSS directly in Ruka source, enabling server-side rendering without a template engine. Components would be ordinary Ruka functions.

Ruka.hdl

A hardware description language using Ruka syntax, targeting FPGA development on SoC boards such as the Xilinx Zynq family. Hardware modules expose getter and setter interfaces, allowing the same language to describe both the software and the hardware it runs on.
