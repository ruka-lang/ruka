Comments

Single-line comments begin with // and run to the end of the line. Block comments are not currently supported.

// This is a comment
let x = 12  // Inline comment

Bindings

Ruka has four binding forms. All bindings are declared with an initial value; there are no uninitialized variables. Bindings can be shadowed by reusing the same identifier, or discarded with _.

KeywordMutabilityVisibility
letMutableScoped to the enclosing block
constImmutablePublic — exported from its module. Static values only — the right-hand side must be a compile-time-known value.
localImmutablePrivate — not exported
testImmutableNot exported — compiled in debug/test builds only
 
let count = 0          // mutable runtime value
count = count + 1

const PI = 3.14159     // public static constant
local max_retries = 5  // private static constant

Functions and types are stored as values in const or local bindings. There is no separate fn or type declaration syntax — the binding keyword determines visibility. Test functions use the test keyword and are only present in debug and test builds.

const is for static values only. The right-hand side of a const binding must be a value known at compile time — a type, a function, a literal, or the result of a compile-time expression. let and local can hold runtime-evaluated values.

Binding Modes

By default variables and non-eva constants are allocated by the GC. Binding modes allows the user to adjust this behaviour. A mode is placed between the binding keyword and the identifier.

ModeSyntaxBehaviour
stc let stc name = ... Stack-allocated. Passed by pointer — not GC-managed and not copied. Can be moved: ownership may be transferred to another binding or function, after which the original is invalid.
eva local eva name = ... / const eva name = ... Forces a runtime function to be evaluated at compile time. All arguments must be compile-time known. The result is stored as a compile-time constant.

For more detail see: Compile-Time Evaluation.

eva on a binding is only needed when you want to force a runtime function to run at compile time. Results from calling functions that already have eva parameters are automatically compile-time — no annotation is required on the binding, but it must be const or local.

Literals

Scalar Literals

let n    = 42          // integer  (isize by default)
let neg  = -7          // signed integer
let hex  = 0xFF        // hexadecimal integer
let bits = 0b00101010  // binary integer
let f    = 3.14        // float   (float by default)
let yes  = true        // boolean
let no   = false

Character Literals

Character literals use single quotes and are inferred as u8. They represent a single ASCII byte. Standard escape sequences are supported. 

let ch  = 'a'    // u8 — 97
let nl  = '\n'   // u8 — 10
let tab = '\t'   // u8 — 9
let nul = '\0'   // u8 — 0

String Literals

Strings use double quotes. String interpolation embeds expressions with ${...}.

let name     = "Ruka"
let greeting = "Hello, ${name}!"

Multiline strings use the pipe-quote form. Each content line begins with |, which is stripped from the resulting string.

let poem = |"
           | Roses are red,
           | Violets are blue.
           |"

Tuple and Array Literals

Tuples are fixed-length, heterogeneous sequences. Arrays are homogeneous sequences. Both use the same dot-brace literal syntax .{...}. The compiler resolves which type to produce using result location semantics: it inspects the expected type from the surrounding context (the left-hand side of a binding, a record field type, or a function parameter type) and produces the appropriate collection.

When there is no type context, .{...} is inferred as a tuple by default. An explicit type annotation on the binding is the most common way to request an array. 

// No type context — inferred as a tuple
let point  = .{10, 20}            // [i32, i32]
let triple = .{1, "hello", true}  // [i32, string, bool]

let y = point[1]

// Explicit array annotation — compiler produces an array
let primes: [i32] = .{2, 3, 5, 7, 11}
let empty:  [i32] = .{}

let two = primes[0]

// Type context from a function parameter resolves the ambiguity
const sum_all = (values: [i32]) -> i32 => // ...

sum_all(.{1, 2, 3})  // .{...} produced as [i32] (array) from parameter type

Explicit Type-Prefixed Collection Literals

Collections can also be created with a type prefix — [type].{...} — which makes the intended type explicit without requiring a binding annotation. This is the idiomatic form for arrays and maps created inline or passed directly to a function. It mirrors the syntax used for record and variant literals. 

// Array — type is explicit, no annotation needed on the binding
let primes = [i32].{2, 3, 5, 7, 11}

// Tuple — multiple types listed
let pair = [i32, string].{42, "hello"}

// Map — key => value type
let scores = [string => i32].{
    "alice" => 95,
    "bob"   => 87
}

Collection Comprehensions

Arrays and maps support comprehension syntax inside the [type].{...} form. A for expression inside the braces generates the elements. 

// Array comprehension — produce 3 random floats
let weights = [f64].{ for 0..3 => random.float(f64, -1.0, 1.0) }

// Array comprehension with an iteration variable
let squares = [i32].{ for i in 1..=5 => i * i }
// .{1, 4, 9, 16, 25}

// Map comprehension — build a map from an array
let arr = [f64].{1.0, 2.0, 3.0, 4.0}
let labels = [string => f64].{ for e in arr => .{"number ${e}" => e} }

Hashmap Literals

Hashmaps use dot-brace notation with => to separate keys from values. This distinguishes them from record literals (which use =). 

let scores = .{
    "alice" => 95,
    "bob"   => 87,
    "carol" => 91
}

local score = scores["alice"]

Record Literals

Record literals use dot-brace notation with = for named fields. If a local variable shares the name of a field, the field can be omitted (shorthand). 

local Vector = record {
    x: f64,
    y: f64
}

let v = Vector.{ x = 1.0, y = 2.0 }

// Field shorthand — variable name matches field name
let x = 3.0
let y = 4.0
let w = Vector.{ x, y }  // equivalent to .{ x = x, y = y }

let x = w.x

Variant Constructors

Variant values are constructed using dot-prefix notation on the variant tag name.

local Color = variant {
    red,
    green,
    blue,
    custom: [u8, u8, u8]
}

let c1 = Color.red
let c2 = Color.custom(.{255, 128, 0})

Option and Result Literals

option(T) and result(T, E) are built-in variant types with their own constructors. Shorthand is provided for these types ?(T) and !(T, E) respectively. 

let some_val: option(i32) = .some(42)
let no_val: ?(i32) = .none

let ok_val: result(i32, string) = .ok(100)
let err_val: !(i32, string) = .err("something went wrong")

Range Literals

The .. and ..= operators produce range values. The exclusive form a..b covers a up to but not including b. The inclusive form a..=b includes both endpoints. 

let exclusive = 0..10    // 0, 1, ..., 9
let inclusive = 0..=10   // 0, 1, ..., 10

// Variable bounds
let start = 2
let end   = 7
for i in start..end do
    @println("${i}")     // 2 3 4 5 6
end

// Slice an array with a range
let nums: [i32] = .{10, 20, 30, 40, 50}
let slice = nums[1..4]   // .{20, 30, 40}

Unwrap Methods

Both built-in variant types provide methods for force-unwrapping their inner value. These methods panic at runtime if the value is absent or an error.

SyntaxTypeBehaviour
expr.?()option(T)Returns the inner T; panics if .none.
expr.!()result(T, E)Returns the inner T; panics if .err.
 
let name = .some("Ruka")
let s = name.?()               // "Ruka" — panics if .none

let res = !(i32, string).ok(42)
let n = res.!()                // 42 — panics if .err

// Common in chains where failure is a programmer error
let count = parse_int(input).!()
let upper = find_user(id).?().name

Range Types

A range value is produced by the .. and ..= operators. Both bounds must be the same numeric type (inferred from context). Ranges implement the iterator interface and can be used in for loops and as array/tuple slice indices.

LiteralBoundsDescription
start..endstart ≤ i < endExclusive range — end value not included
start..=endstart ≤ i ≤ endInclusive range — end value included
 
for i in 0..5 do
    @println("${i}")     // 0 1 2 3 4
end

for i in 1..=5 do
    @println("${i}")     // 1 2 3 4 5
end

// Ranges as slice indices
let nums: [i32] = .{10, 20, 30, 40, 50}
let middle = nums[1..4]   // .{20, 30, 40}
let tail   = nums[2..=4]  // .{30, 40, 50}

Type Annotations

Type annotations follow a colon after the identifier. They are never required where the type can be inferred — see Type Inference for the full rules. Annotations are only written to resolve genuine ambiguity or to improve readability at module boundaries. 

let x    = 10              // inferred as isize — default integer type
let f    = 3.14            // inferred as float  — default float type
let name = "Ruka"          // inferred as string

let y: u8  = 10            // annotation required — non-default integer width
let g: f32 = 3.14          // annotation required — non-default float width

let scores: [f64] = .{9.5, 8.0, 7.3}   // annotation required — collection literal needs context
let pair: [i32, string] = .{1, "one"}
let table: [string => usize] = .{ "a" => 1 }

User-defined Types

Types are values stored in const or local bindings. A type definition describes the memory layout only — behaviour (methods and static functions) is declared separately.

Records

Records are product types (similar to structs). Fields are separated by commas. A field can be made private with the local keyword.

local Vector = record {
    x: f64,
    y: f64
}

local User = record {
    name:  string,
    email: string,
    local password_hash: string  // private field
}

// Inline (anonymous) record types are also valid
let origin: record { x: f64, y: f64 } = .{ x = 0.0, y = 0.0 }

Record Destructuring

A record value can be destructured into individual bindings using the .{…} syntax on the left-hand side of a binding. The binding keyword determines visibility and mutability of the resulting names. This is the same mechanism used when destructuring imported files. 

local Point = record { x: f64, y: f64 }

let p = Point.{ x = 3.0, y = 4.0 }

// Destructure into separate bindings
local .{ x, y } = p
// x is 3.0, y is 4.0

// Works with any record — including imported files
local .{ sqrt, pow } = @import("math")
local .{ Dog, Cat }  = @import("animals")

Variants (Tagged Unions)

Variants are sum types. Each tag can optionally carry a payload. A tag with no payload has the unit type. 

local Shape = variant {
    circle:    f64,            // payload: radius
    rectangle: [f64, f64],    // payload: width, height
    point                      // no payload
}

// Variants can be nested or reference other types
local Tree = variant {
    leaf: i32,
    node: record { left: Tree, right: Tree }
}

// Inline variants inside a record field
local Cat = record {
    name: string,
    colour: variant {
        black,
        white,
        grey
    }
}

Type Inference

Ruka uses aggressive bidirectional type inference. Type annotations are only required to resolve genuine ambiguity — the compiler reports a hard error when it cannot determine a unique type. There is no implicit fallback type and no silent coercion.

Methods and static functions must be declared in the same scope or module as their type. This is what makes inference possible: the compiler always knows the complete set of types to consider when resolving an unannotated expression.

Method Receiver Type Inference

Because self is a keyword, the compiler always knows a declaration is a method. What it still needs to determine is which type self belongs to. If self.fieldName is accessed and exactly one type in scope has that field, the receiver type is resolved without an annotation. When the field exists on more than one type, an explicit annotation (self: TypeName) is required — and the compiler will tell you. 

local Circle = record { radius: f64 }
local Rect   = record { width: f64, height: f64 }

// self inferred as Circle — only type in scope with `radius`
// return type inferred as f64 from the expression
const area (self) = () => 3.14159 * self.radius * self.radius

// self inferred as Rect — only type with both `width` and `height`
const area (self) = () => self.width * self.height

local Dog = record { name: string, breed: string }
local Cat = record { name: string, indoor: bool }

// Inferred as Dog — only Dog has both `name` and `breed`
const label (self) = () => "${self.name} (${self.breed})"
// Explicit annotation required — both types have a `name` field
const label (self: Cat) = () => "${self.name}"

Record Literal Inference

A record literal .{ field = value, ... } infers its type from the set of fields provided. If exactly one type in scope has that field set, no type prefix is needed. This applies to bindings, function arguments, and return expressions. 

local Circle = record { radius: f64 }
local Rect   = record { width: f64, height: f64 }

// Inferred as Circle — only type with `radius`
let c = .{ radius = 5.0 }

// Inferred as Rect — only type with `width` and `height`
let r = .{ width = 4.0, height = 6.0 }

local Dog = record { name: string, breed: string }
local Cat = record { name: string, indoor: bool }

let d  = .{ name = "Rex",      breed = "Labrador" }  // Dog — Cat has no `breed`
let c2 = .{ name = "Whiskers", indoor = true }        // Cat — Dog has no `indoor`

Return Type Inference

Return types are inferred from the last expression in a block or from the expression after =>. Explicit return type annotations are only needed when the inferred type is ambiguous, or as documentation at a public API boundary. 

// Return type inferred as f64
const circumference (self) = () => 2.0 * 3.14159 * self.radius

// Return type inferred as string — interpolation always produces string
const label (self) = () => "Circle(r=${self.radius})"

// Return type inferred as option(i32) from the .some / .none arms
const find = (values: [i32], target: i32) do
    for i in 0..@len(values) do
        if values[i] == target do return .some(i) end
    end
    .none
end

Numeric Literal Defaults

Numeric literals have default types that apply when no context provides a more specific type.

LiteralDefault typeOverride example
Integer (42, -7, 0xFF) isize let x: u8 = 42
Float (3.14, -0.5) float let x: f32 = 3.14 
let a = 42          // isize
let b = 3.14        // float

// Context overrides the default
let c: u8  = 42    // u8  — annotation provides context
let d: f64 = 3.14  // f64 — annotation provides context

// Function parameter type provides context
const double = (x: i32) => x * 2
let result = double(10)  // 10 is produced as i32, not isize
Ambiguity is always a compile error. The compiler will highlight the ambiguous expression and report which types are candidates. Resolve it by adding a type annotation at the point of ambiguity.

Control Flow

All control flow constructs in Ruka are expressions — they produce a value. Logical operators use keywords: and, or, not.

If / Else

Multiline branches use do…end. A single-expression branch uses =>. If an if expression is used as a value, all branches must return the same type. 

// Multiline
if x > 0 do
    @println("positive")
else if x < 0 do
    @println("negative")
else do
    @println("zero")
end

// Single-line — => introduces a single expression; no end required
if x > 0 => @println("positive")

// Single-line with else
if x > 0 => @println("positive") else => @println("non-positive")

// Single-line chained
if x > 0 => "positive" else if x < 0 => "negative" else => "zero"

// Used as a value (do...end form)
let label = if score >= 90 do
    "A"
else if score >= 80 do
    "B"
else
    "C"
end

// Logical operators
if condition and not other_condition do
    // ...
end

if a or b do
    // ...
end

Bitwise Operators

Bitwise operators work on integer types. They operate on the raw bit representation of the value. 

let a = 0b1100   // 12
let b = 0b1010   // 10

let and_result = a & b    // 0b1000 — 8
let or_result  = a | b    // 0b1110 — 14
let xor_result = a ^ b    // 0b0110 — 6

Exponentiation

The ** operator raises the left operand to the power of the right. It is right-associative, so 2 ** 3 ** 2 evaluates as 2 ** (3 ** 2). 

let squared = 4 ** 2     // 16
let cubed   = 2 ** 3     // 8

// Right-associative
let nested = 2 ** 3 ** 2   // 2 ** 9 = 512

Pattern Matching

The match expression compares a value against a series of arms. Arms are tested in order. The result of a match is the value of the matched arm.

Multiline arms use do…end. Single-expression arms use =>. The else arm is the catch-all default.

Matching literals and conditions

match status_code with
    200 => @println("OK")
    404 => @println("Not Found")
    500 do
        @println("Internal Server Error")
        log_error()
    end
    else => @println("Unknown status")
end

Matching and binding variant payloads

local Shape = variant {
    circle:    f64,
    rectangle: [f64, f64],
    point
}

let area = match shape with
    .circle(r)           => 3.14159 * r * r
    .rectangle(.{w, h})  => w * h
    .point               => 0.0
end

// option(T)
match maybe_name with
    .some(name) => @println("Hello, ${name}")
    .none       => @println("No name provided")
end

Guard conditions

Arms may use boolean conditions involving the matched value or any in-scope variable.

match n with
    0       => "zero"
    n < 0   => "negative"
    n > 100 => "large"
    else    => "other"
end

Loops

While Loop

let i = 0
while i < 10 do
    @println("${i}")
    i = i + 1
end

// Single-expression body — no end required
while i < 10 => i = i + 1

Destructuring Conditions

A pattern match can serve as the condition of an if or while using the pattern = expr form. The condition is true only if the pattern matches, and any bindings it introduces are in scope for the body. 

// if with destructuring — body only runs if the option is .some
let maybe: option(i32) = .some(42)
if .some(x) = maybe do
    @println("got ${x}")
end

// while with destructuring — loop until the value is .none
while .some(x) = next_value() do
    process(x)
end

For Loop

Iterates over any value that implements the iterator interface. The most common iterable is a range literal. 

// Range — the common numeric loop idiom
for i in 0..5 do
    @println("${i}")     // 0 1 2 3 4
end

// Single-expression body — no end required
for i in 0..5 => @println("${i}")

// Range without a binding variable
for 0..10 => @println("tick")

// Inclusive range
for i in 1..=3 do
    @println("${i}")     // 1 2 3
end

// Iterating over an array
let numbers = .{10, 20, 30}
for n in numbers do
    @println("${n}")
end

// Destructuring a tuple in a for loop
let pairs: [[i32, string]] = .{ .{1, "one"}, .{2, "two"}, .{3, "three"} }

for .{k, v} in pairs do
    @println("${k} = ${v}")
end

Pattern-Filtered For Loop

A pattern in the for loop variable filters the sequence: only elements that match the pattern are visited. Non-matching elements are skipped silently. 

let values: [option(i32)] = .{.some(1), .none, .some(3), .none, .some(5)}

// Only visits elements where the option is .some; x is bound to the inner value
for .some(x) in values do
    @println("${x}")    // 1, 3, 5
end

Break

break immediately exits the innermost enclosing while or for loop. 

// Exit a while loop early
let i = 0
while true do
    if i >= 5 do break end
    @println("${i}")
    i = i + 1
end

// Stop searching once a match is found
let found = false
for item in items do
    if item == target do
        found = true
        break
    end
end

Continue

continue skips the remainder of the current loop iteration and moves immediately to the next one. 

// Print only odd numbers
for i in 0..10 do
    if i % 2 == 0 do continue end
    @println("${i}")     // 1 3 5 7 9
end

// Skip invalid entries while processing
for entry in entries do
    if not entry.is_valid() do continue end
    process(entry)
end

Defer

A defer statement schedules an expression to run at the end of the enclosing do…end block, regardless of how control exits that block — whether by reaching the end, an early return, or a break. This makes defer the idiomatic way to pair cleanup with acquisition.

When multiple defer statements appear in the same block they execute in LIFO order — the last one declared runs first. 

// Guaranteed cleanup: close() always runs when the block exits
const read_file = (path: string) do
    let file = open(path)
    defer file.close()

    // ... work with file
end   // file.close() is called here

// Multiple defers execute in reverse declaration order
const example = () do
    defer @println("third")    // runs last
    defer @println("second")
    defer @println("first")    // runs first
    // Output on exit: first, second, third
end

// Defer works at any scope depth — it binds to the nearest enclosing block
const process = (items: [i32]) do
    let resource = acquire()
    defer resource.release()       // released when process() returns

    for item in items do
        let handle = item_handle(item)
        defer handle.close()       // closed at the end of each loop iteration
        handle.transform()
    end
end
defer captures variables by reference at the point of declaration. The deferred expression sees the variable values as they are when the block exits, not when defer was written.

Functions

Functions are anonymous values stored in bindings. There is no special fn keyword — a function is written as a parameter list followed by a body block or single expression.

The last expression in a block is returned. An early return can be written with the return keyword.

Multiline function (do…end block)

local add = (a: i32, b: i32) do
    a + b
end

local greet = (name: string) do
    let msg = "Hello, ${name}!"
    @println(msg)
    // last expression is (), the unit value
end

Single-expression function (arrow)

The => form means the body is the rest of that same line — no closing end is required. For bodies that span multiple lines, use do…end. 

local square = (x: i32) => x * x

local abs = (x: i32) => if x < 0 do -x else x end

// Multi-line body — must use do...end
local max = (a: i32, b: i32) do
    if a > b do a
    else b
    end
end

Return type annotation

A -> after the parameter list specifies the return type. This applies whether the body is a same-line => or a do…end block. 

// Inline — return type after the parameter list
local square = (x: i32) -> i32 => x * x

// Works identically with do...end
local clamp = (x: i32, lo: i32, hi: i32) -> i32 do
    if x < lo do lo else if x > hi do hi else x end
end

// Binding-side annotation — function type written as parameter list then return type
local square2: (i32) -> i32 = (x) => x * x

Automatic Return Wrapping

When a function's return type is annotated as option(T) or result(T, E), the compiler automatically wraps returned values rather than requiring explicit .some(...) or .ok(...) at every return site. The wrapping is inferred from the type of the returned expression:

  • option(T): a T value is wrapped as .some(T); a () expression becomes .none.
  • result(T, E): a T value is wrapped as .ok(T); an E value is wrapped as .err(E).
// Returns option(usize) — wrapping is automatic
const find_index = (arr: [i32], target: i32) -> option(usize) do
    for i in 0..@len(arr) do
        if arr[i] == target do return i end   // i (usize) → .some(i)
    end
    // () at end of block → .none
end

// Returns result(f64, string) — wrapping is automatic
const safe_div = (a: f64, b: f64) -> result(f64, string) do
    if b == 0.0 do return "division by zero" end   // string → .err(...)
    a / b                                           // f64   → .ok(...)
end

Entry point

The program entry point is a public constant named main.

const main = () do
    @println("Hello, world!")
end

Static Functions

A static is any value associated with a type using the receiver syntax — a type name in parentheses between the binding name and the = sign. Any value can be attached as a static, not just functions: constants, default instances, and lookup tables are all valid. Using a type name in the receiver position — rather than the reserved keyword self — is what distinguishes a static from a method. 

local Vector = record {
    x: f64,
    y: f64
}

// Public static function — called as Vector.new(...)
const new (Vector) = (x, y) do
    Vector.{ x, y }
end

// Private static function
local zero (Vector) = () => Vector.{ x = 0.0, y = 0.0 }

const main = () do
    let v      = Vector.new(1.0, 2.0)
    let origin = Vector.zero()
end

Non-Function Statics

Any compile-time value — a constant, a default instance, a type — can be attached as a static. Access uses the same TypeName.name dot notation. 

local Color = record { r: u8, g: u8, b: u8 }

// Constant values attached as statics
const RED   (Color) = Color.{ r = 255, g = 0,   b = 0   }
const GREEN (Color) = Color.{ r = 0,   g = 255, b = 0   }
const BLUE  (Color) = Color.{ r = 0,   g = 0,   b = 255 }

const main = () do
    let c = Color.RED
    @println("r=${c.r}")   // r=255
end

Static Functions on Variants

Static functions can be attached to variant types using the same receiver syntax. This is useful for named constructors and factory helpers. 

local Direction = variant {
    north,
    south,
    east,
    west
}

const from_str (Direction) = (s) do
    match s with
        "north" => .some(Direction.north)
        "south" => .some(Direction.south)
        "east"  => .some(Direction.east)
        "west"  => .some(Direction.west)
        else    => .none
    end
end

const main = () do
    let d = Direction.from_str("east")
end
Scope restriction: Static functions and methods may only be declared in the same scope or module where the type is defined. This is required for type inference — the compiler resolves receiver types and record literals by checking which types are in the current scope. Attaching behaviour from a foreign module would make that resolution ambiguous.

Methods

A method is declared like a static function, but uses the reserved keyword self in the receiver position instead of a type name. self refers to the instance the method is called on and is always the first implicit argument. Any parameter mode (mut, mov, etc.) may precede self.

This is the syntactic distinction between a method and a static function: if the receiver is self it is a method; if it is a type name it is a static function. Because self is a keyword, the difference is unambiguous regardless of naming conventions. 

local Vector = record {
    x: f64,
    y: f64
}

// Public method — self is the receiver instance; its type is inferred as Vector
const add (self) = (other) do
    Vector.{ x = self.x + other.x, y = self.y + other.y }
end

const length (self) = () do
    @sqrt(self.x * self.x + self.y * self.y)
end

// Private method
local negate (self) = () => Vector.{ x = -self.x, y = -self.y }

const main = () do
    let a = Vector.new(1.0, 0.0)
    let b = Vector.new(0.0, 1.0)
    let c = a.add(b)
    @println("Length: ${c.length()}")
end

Mutable Methods

To mutate the receiver inside a method, apply the mut mode before self. The caller must hold a mutable binding. 

local Counter = record { value: i32 }

const new (Counter) = () => Counter.{ value = 0 }

// mut self — the receiver is a mutable reference
const increment (mut self) = () do
    self.value = self.value + 1
end

const main = () do
    let c = Counter.new()
    c.increment()         // value is now 1
    c.increment()         // value is now 2
    @println("${c.value}")
end

Methods on Variants

Methods can also be attached to variant types using the same self receiver syntax. Pattern matching on self inside the body handles each tag. 

local Direction = variant {
    north,
    south,
    east,
    west
}

const opposite (self) = () do
    match self with
        .north => Direction.south
        .south => Direction.north
        .east  => Direction.west
        .west  => Direction.east
    end
end

const is_vertical (self) = () =>
    self == Direction.north or self == Direction.south

const main = () do
    let d = Direction.north
    @println("${d.opposite().is_vertical()}")  // true
end

Parameter Modes

By default, all arguments are passed as immutable references managed by the garbage collector. A mode keyword placed before the parameter identifier changes how the compiler treats the lifetime and ownership of that value. Modes apply to any parameter, including method receivers. Some are shared with binding modes.

ModeSyntaxBehaviour
mut (mut x: T) Mutable reference. Changes to x are visible to the caller.
mov (mov x: T) Ownership is transferred into this function. The caller cannot use the value after the call.
stc (stc x: T) Stack-allocated pass-by-copy.
eva (eva x: T) or (x: type) Compile-time constant. The value must be known at compile time. Inferred automatically when the parameter type is type. 
// mut — in-place mutation
const fill (mut self) = (value: u8) do
    // modifies caller's buffer
end

// mov — consume / transfer ownership
const close = (mov handle: FileHandle) do
    // handle is invalid for the caller after this call
end

// stc — stack allocated pass-by-copy
const hash = (stc data: [u8]) -> u64 do
    // data is on the stack, cannot be returned without a copy
end

// eva — compile-time value (explicit annotation)
const repeat = (eva times: usize, msg: string) do
    let i = 0
    while i < times do
        @println(msg)
        i = i + 1
    end
end
Note: eva is covered in detail in the Compile-Time Evaluation section, including automatic inference from type-annotated parameters and storing compile-time results.

Named Parameters

Prepending ~ to a parameter name makes it a named parameter. Named parameters must be passed by label at the call site using ~label=value syntax and may appear in any order. Positional and named parameters can be freely mixed.

Declaring Named Parameters

// Both parameters are named — order at call site doesn't matter
const greet = (~name: string, ~greeting: string) =>
    "${greeting}, ${name}!"

// Mix of positional and named
const clamp = (x: isize, ~lo: isize, ~hi: isize) do
    if x < lo do lo else if x > hi do hi else x end
end

Calling with Named Parameters

greet(~name="Ruka", ~greeting="Hello")   // "Hello, Ruka!"
greet(~greeting="Hi", ~name="World")     // "Hi, World!" — order doesn't matter

let v = clamp(42, ~lo=0, ~hi=10)         // 10 — positional first, named in any order

Label Shorthand

If a local variable shares the label name, the =value part can be omitted. 

let name     = "Ruka"
let greeting = "Hello"

greet(~name, ~greeting)   // shorthand for greet(~name=name, ~greeting=greeting)

Optional Named Parameters

A named parameter whose type is option(T) is optional at the call site. When the argument is omitted, the parameter receives .none automatically. When a value is supplied, it is automatically wrapped in .some(...) — no explicit wrapping is needed. 

const greet = (~name: string, ~title: option(string)) =>
    if .some(t) = title do "${t} ${name}" else name end

greet(~name="Ruka")                    // "Ruka"          — title is .none
greet(~name="Ruka", ~title="Dr.")      // "Dr. Ruka"      — title is .some("Dr.")
greet(~name="Ruka", ~title=.none)      // explicit .none also valid

Trailing Block Syntax

A named parameter can be passed after the closing parenthesis as a trailing argument. This is most useful for higher-order functions that accept a closure — the named argument reads as a block visually attached to the call. Only one trailing argument is permitted per call. 

const apply_twice = (x: isize, ~f: (isize) -> isize) => f(f(x))

// Named param inside the argument list — standard form
let r1 = apply_twice(3, ~f=(x) => x * 2)    // 12

// Trailing form — same result, reads like an attached block
let r2 = apply_twice(3) ~f=(x) => x * 2     // 12

// Multi-line trailing block
let r3 = apply_twice(3) ~f=(x) do
    let y = x * 2
    y + 1
end                                           // 14

Higher-order functions

The trailing syntax is especially readable for collection operations. Named closure parameters are placed last in the definition so callers can trail them. 

// `t: type` and `u: type` infer eva automatically
const map    = (t: type, u: type, arr: [t], ~f: (t) -> u) -> [u] do ... end
const filter = (t: type, arr: [t], ~pred: (t) -> bool) -> [t] do ... end
const reduce = (t: type, u: type, arr: [t], init: u, ~f: (u, t) -> u) -> u do ... end

let nums = .{1, 2, 3, 4, 5}

let doubled = map(nums) ~f=(x) => x * 2
let evens   = filter(nums) ~pred=(x) => x % 2 == 0
let total   = reduce(nums, 0) ~f=(acc, x) => acc + x   // 15

// Multi-line body — the trailing block scales naturally
let results = map(nums) ~f=(x) do
    let squared = x * x
    squared - 1
end
Note: Named parameters are always identified by label, never by position. A named parameter cannot be passed without its ~label= prefix.

Interfaces

An interface defines a set of method signatures that a type must satisfy. Interfaces describe behaviour only — they carry no fields or state. Any type that implements all of an interface's methods automatically satisfies it; there is no explicit implements declaration.

Inside the braces, each line declares one method signature: the method name, its receiver (with optional mode), then the remaining parameter list and return type.

Declaring an Interface

// A type that can be printed
const Printable = interface {
    print(self): () -> ()
}

// A type that supports a binary add operation
const Adder = interface {
    add(mut self): (b: usize) -> ()
}

// Multiple methods
const Shape = interface {
    area(self):      () -> f64
    perimeter(self): () -> f64
}

Satisfying an Interface

A type satisfies an interface implicitly. As long as the required methods are declared in the same module as the type, the compiler considers the type to implement the interface. Receiver and return types are typically inferred — annotations are only needed when the field set is ambiguous. 

local Circle = record { radius: f64 }

// self inferred as Circle (only type in scope with `radius`);
// return types inferred from the expressions
const area      (self) = () => 3.14159 * self.radius * self.radius
const perimeter (self) = () => 2.0 * 3.14159 * self.radius

// Circle now satisfies the Shape interface — no declaration needed

Dynamic Dispatch via Interface Types

When a parameter's type is an interface, dynamic dispatch is inferred automatically. Any value whose type satisfies the interface can be passed. Interface types are never inferred — they must always be written explicitly as a type annotation. 

// Interface type annotation on a parameter — dynamic dispatch is automatic
const print_shape_info = (s: Shape) do
    @println("area:      ${s.area()}")
    @println("perimeter: ${s.perimeter()}")
end

local Rect = record { w: f64, h: f64 }

const area      (self) = () => self.w * self.h
const perimeter (self) = () => 2.0 * (self.w + self.h)

const main = () do
    let c = Circle.{ radius = 5.0 }
    let r = Rect.{ w = 4.0, h = 6.0 }

    print_shape_info(c)   // Circle satisfies Shape
    print_shape_info(r)   // Rect satisfies Shape
end
Note: Interfaces specify only methods. Fields cannot appear in an interface definition. To share field access across types, expose it through methods.

Pipelines

The pipeline operator |> passes the left-hand value as the first argument to the right-hand function. It allows chaining operations in the order they are applied, which is easier to read than deeply nested calls. 

// Without pipeline
@println(trim(toUpperCase(getName(user))))

// With pipeline — reads top to bottom, left to right
getName(user)
    |> toUpperCase()
    |> trim()
    |> @println()

The pipeline works with any function — free functions, static functions, and methods called in function position. It always inserts the left-hand value as the first argument. 

local Dog = record { name: string, breed: string }

const about (self) = () => "I'm ${self.name}, a ${self.breed}."

const main = () do
    let ruka = Dog.{ name = "Ruka", breed = "Jack Russell Terrier" }

    // Method call followed by pipeline into a built-in
    ruka.about()
        |> @println()
end

Modules

A file is a record. There is no separate module concept — every Ruka source file implicitly defines a compile-time record value whose fields are the file's const bindings. local bindings at file scope are private fields: they are accessible within the file but not from any importer.

Because files are records, all of the language's record machinery applies directly: field access retrieves exports, and destructuring pulls named fields into scope. The local keyword marks privacy consistently whether on a record field or a file-scope binding.

A file record is an empty record — it has no instance fields, only statics. It cannot be instantiated with .{}. Every declaration in the file is stored as a static field accessible via the record value returned by @import. 

// stack.ruka — a file is a record; const bindings are its public fields

local t = record {         // local binding — private to this file
    local data: [i32],
    local len:  usize
}

const new (t) = () => .{ data = .{}, len = 0 }   // public field

const push (mut self) = (value: i32) do           // public field
    // append value to self.data
    self.len = self.len + 1
end

const pop (mut self) = () -> option(i32) do       // public field
    if self.len == 0 do
        .none
    else do
        self.len = self.len - 1
        .some(self.data[self.len])
    end
end

Imports

@import evaluates at compile time and returns the file's record value. The result is always compile-time known — eva is implied. Accessing an export is ordinary field access; destructuring an import is ordinary record destructuring, using the same .{…} syntax. 

// Field access on the imported record
local math = @import("math")

const main = () do
    @println("${math.sqrt(2.0)}")
end

// Record destructuring — pull specific fields into scope
local .{ sqrt, pow } = @import("math")
local .{ Dog, Cat }  = @import("animals")

// Imported records can be passed to eva parameters like any compile-time value
const describe = (t: type) => @println("${t}")
local animals = @import("animals")

const main = () do
    describe(animals.Dog)   // passes the Dog type to a compile-time parameter
end

Test Bindings

The test keyword declares a test function. Test bindings follow the same syntax as any other binding, but with two important differences: the value must be a zero-argument function, and the binding is compiled only in debug and test builds. In optimised builds test declarations are elided entirely — they contribute no code and no overhead to production binaries.

Tests are never exported. They can access everything in their module's scope, including local (private) bindings, which makes it straightforward to test internal implementation details without exposing them. 

local add = (a: i32, b: i32) => a + b

test addition = () do
    @assertEq(add(1, 2), 3)
    @assertEq(add(-1, 1), 0)
    @assertEq(add(0, 0), 0)
end

test addition_is_commutative = () do
    @assertEq(add(3, 4), add(4, 3))
end

Because tests live in the same scope as the module, they have access to local (private) definitions directly — no special test-visibility mechanism is needed. 

// Private implementation detail — not exported
local clamp_internal = (x: i32, lo: i32, hi: i32) => if x < lo do lo else if x > hi do hi else x end

test clamp_lower_bound = () do
    @assertEq(clamp_internal(3, 5, 10), 5)
end

test clamp_upper_bound = () do
    @assertEq(clamp_internal(15, 5, 10), 10)
end

test clamp_in_range = () do
    @assertEq(clamp_internal(7, 5, 10), 7)
end
Compilation modes. test bindings are included in two build modes: debug (full code coverage analysis, all assertions active) and test (runs the test suite and reports pass/fail). They are stripped in the release build. This means @assertEq and other assertion calls inside a test binding never appear in a production binary.

Built-in Functions

Built-in functions begin with @ to distinguish them from user-defined names. They are available in all scopes without any import.

FunctionDescription
@print(string)Print a string to standard output without a trailing newline.
@println(string)Print a string to standard output followed by a newline.
@import(path)Import a file as a module. Returns the module value.
@typeOf(value)Return the compile-time type of a value.
@abs(number)Absolute value.
@sqrt(number)Square root.
@assertEq(a, b)Assert that two values are equal. Panics on failure.
@len(collection)Returns the length of an array, tuple, or string as usize.

@now()Returns the current time as a timestamp value.
 
@println("Hello!")                       // Hello!
@println("${@typeOf(42)}")               // isize
@assertEq(@abs(-5), 5)

Compile-Time Evaluation

Ruka includes a compile-time interpreter. Functions with eva parameters are evaluated at compile time rather than at runtime. This enables generic programming through static dispatch and the generation of complex types that cannot be expressed as simple literals. Types, functions, and modules are first-class values in compile-time contexts.

Automatic eva Inference

When a parameter's type annotation is type, the eva mode is inferred automatically — no explicit keyword is needed. Writing (t: type) is exactly equivalent to (eva t: type). When a parameter is used as a type annotation for another parameter, both the eva mode and the type annotation are inferred. 

// Explicit — valid, but redundant
const min = (eva t: type, a: t, b: t) => if a < b do a else b end

// Inferred — preferred; `eva` is implied by the `type` annotation
const min = (t: type, a: t, b: t) => if a < b do a else b end

// Both `eva` and `: type` inferred — `t` is used to annotate `a` and `b`
const min = (t, a: t, b: t) => if a < b do a else b end

// Compile-time type constructor — same rules apply
const Pair = (a, b) do
    record { first: a, second: b }
end

Generics via eva

A function with a type-annotated parameter is instantiated separately for each unique set of compile-time arguments, similar to monomorphisation in Rust or comptime in Zig. 

const min = (t, a: t, b: t) => if a < b do a else b end

const main = () do
    local x = min(i32, 3, 7)     // compiler emits an i32 version
    local y = min(f64, 1.5, 2.0) // compiler emits an f64 version
end

Generating Types at Compile Time

Compile-time functions can return types, enabling parameterised type definitions. Non-type parameters that must be compile-time constants still require an explicit eva annotation. 

// Returns a record type — `cap` requires explicit eva because it is not type-annotated as `type`
const FixedArray = (t, eva cap: usize) do
    record {
        data: [t],
        len:  usize
    }
end

// Instantiate at compile time — these are concrete types
local IntBuf   = FixedArray(i32, 64)
local FloatBuf = FixedArray(f64, 16)

const main = () do
    let buf = IntBuf.{ data = .{}, len = 0 }
end

Storing Compile-Time Results

If a function has any eva parameters, it runs at compile time and its result is automatically a compile-time constant. The binding must be const or local — assigning a compile-time result to a let binding is a compile error. Results can also be used inline without a binding. 

const Pair = (a, b) do
    record { first: a, second: b }
end

// Binding must be const or local
local IntStrPair = Pair(i32, string)

// let IntStrPair = Pair(i32, string)  // compile error — mutable binding

// Used inline without a binding — also valid
let p = Pair(i32, string).{ first = 1, second = "one" }

Forcing Runtime Functions at Compile Time

A binding declared with eva forces any function — including ordinary runtime functions — to execute at compile time. All arguments must be compile-time known. This is distinct from calling a function that already has eva parameters, which runs at compile time automatically. 

const PI = 3.14159265358979

// @sqrt is a runtime function, but eva forces it to run at compile time
local eva SQRT_2: f64 = @sqrt(2.0)          // literal — compile-time known
local eva INV_PI: f64 = 1.0 / PI            // const PI — compile-time known

// Eva results are themselves compile-time known and can be chained
local eva DOUBLE_SQRT_2 = SQRT_2 * 2.0

// Useful for any expensive or non-trivial constant
local eva LOOKUP = generate_lookup_table(256)
local eva KEY_HASH = compute_hash("app-secret")

Compile-Time Reflection

@typeOf returns the type of any expression as a compile-time value, which can be passed to eva parameters or compared at compile time. 

const assertSameType = (a: type, b: type) do
    if not (a == b) do
        @compileError("type mismatch: expected ${a}, got ${b}")
    end
end

const main = () do
    let x: i32 = 10
    let y: i32 = 20
    assertSameType(@typeOf(x), @typeOf(y))  // ok — both i32
end
Note: Only values that are provably known at compile time may be passed to eva parameters. Passing a runtime value to an eva parameter is a compile error. Compile-time known values include: literals, const and local constants, and results of other compile-time evaluations.
