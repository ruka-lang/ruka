Comments

Single-line comments begin with // and run to the end of the line. Block comments are not currently supported.

// This is a comment
let x = 12  // Inline comment

Bindings

All runtime and compile-time values are introduced with let. Bindings are always initialized — there are no uninitialized variables. A binding can be shadowed by reusing the same identifier, or discarded with _. Mutability and evaluation time are expressed through modes; privacy is determined by the case of the binding name (see Privacy). There is no separate binding keyword for each axis. Test functions use the test keyword (see Test Bindings).

The shape of a binding is:

let [mode]name [: type] = value

The mode prefix is optional; the defaults depend on where the binding appears.

Defaults by scope

ScopeDefault mutabilityDefault evaluation timeDefault visibility
File (top level)ImmutableCompile-time (inferred)By name case
Method / static / type declarationImmutableCompile-time (inferred)By name case
Inner scope (function body, block)ImmutableRuntimen/a (not exported)

Privacy: identifier case

Visibility is determined by the first let ter of the binding name — the inverse of Go's convention. A name beginning with a lowercase let ter (or underscore followed by a lowercase let ter) is public and exported from the file record; a name beginning with an uppercase let ter is private and not exported. The same rule applies to methods, statics, and record or variant fields. Inside a function body, privacy does not apply — name case is purely a convention there. 

let pi = 3.14159             // public static constant (file scope)
let MaxRetries = 5           // private static constant

let main = () do             // public entry point
    ruka.println("Hello")
end

let Helper = () do           // private function
    // ...
end

Mutability: *

A plain let binding is immutable. To make it a mutable variable, prefix the name with *. Reassignment (name = expr) is only permitted on bindings declared with *. 

let x = 12         // immutable
// x = 13          // error — x was not declared with *

let *count = 0     // mutable
count = count + 1  // ok

Binding Modes

A mode prefix is placed directly before the binding identifier with no space between them. Modes let you adjust how a binding is stored or evaluated.

PrefixSyntaxBehaviour
* let *name = ... Mutable variable. The binding may be reassigned; without * a let is immutable.
& let &name = ... Move-on-capture. A runtime binding declared with & has its ownership transferred into any closure that captures it; after capture the original binding is invalid and further use is a compile error. Valid on runtime bindings only — it has no meaning at compile time.
$ let $name = ... Stack-allocated. Passed by pointer — not GC-managed and not copied. Can be moved: ownership may be transferred to another binding or function, after which the original is invalid.
@ let @name = ... Forces the right-hand side to be evaluated at compile time. Required only inside inner scopes, where the default is runtime evaluation; at file scope and on methods/statics @ is inferred and writing it is redundant.

The & (move) mode is valid on parameters and on runtime bindings. Because it only takes effect when a runtime value is captured by a closure, it has no meaning on compile-time (@) bindings. 

let run = () do
    let &buffer = open_buffer()
    let emit = () do buffer.write("log")   // closure captures buffer — ownership moves in
    emit()
    // buffer.close()   // error — buffer was moved into emit
end
Why @ is almost always inferred. A Ruka file is not a script — its top level cannot contain executable statements, only declarations. Every file-level binding is therefore a static whose right-hand side is evaluated at compile time. The same applies to methods, statics, and type declarations. @ only becomes meaningful inside a function body, where you can use it to force an otherwise-runtime computation to be resolved at compile time: 

let run = () do
    let @table = build_lookup_table()   // evaluated at compile time
    let rows   = fetch_rows()           // evaluated at runtime
    // ...
end

For more detail see: Compile-Time Evaluation.

Literals

Scalar Literals

let n    = 42          // integer  (int by default)
let neg  = -7          // signed integer
let hex  = 0xFF        // hexadecimal integer
let bits = 0b00101010  // binary integer
let f    = 3.14        // float   (float by default)
let yes  = true        // boolean
let no   = false

Character Literals

Character literals use single quotes and are inferred as u8. They represent a single ASCII byte. Standard escape sequences are supported. 

let ch = 'a'     // u8 — 97
let nl = '\n'    // u8 — 10
let tab = '\t'   // u8 — 9
let nul = '\0'   // u8 — 0

String Literals

Strings use double quotes. String interpolation embeds expressions with ${...}.

let name     = "Ruka"
let greeting = "Hello, ${name}!"

Multiline strings use the pipe-quote form. Each content line begins with |, which is stripped from the resulting string.

let poem = |"
           | Roses are red,
           | Violets are blue.
           |"

Tuple and Array Literals

Tuples are fixed-length, heterogeneous sequences. Arrays are homogeneous sequences. Both use the same dot-brace literal syntax .{...}. The compiler resolves which type to produce using result location semantics: it inspects the expected type from the surrounding context (the left-hand side of a binding, a record field type, or a function parameter type) and produces the appropriate collection. 

// No type context — compiler error about providing context
let point  = .{10, 20}
// No type context - compiler can inferred tuple because of heterogeneous elements
let triple = .{1, "hello", true} // [int, string, boolean]

let greeting = triple[1]

// Explicit annotation — compiler produces an array
let primes: [i32] = .{2, 3, 5, 7, 11}
let empty: [i32] = .{}

let two = primes[0]

// Type context from a function parameter resolves the ambiguity
let sum_all = (values: [i32]) -> i32 do // ...

sum_all(.{1, 2, 3})  // .{...} produced as [i32] (array) from parameter type

Explicit Type-Prefixed Collection Literals

Collections can also be created with a type prefix — [type].{...} — which makes the intended type explicit without requiring a binding annotation. This is the idiomatic form for creating a new binding initialized to an array or tuple. It mirrors the syntax used for record and variant literals. 

// Array — type is explicit, no annotation needed on the binding
let primes = [i32].{2, 3, 5, 7, 11}

// Tuple — multiple types listed
let pair = [i32, string].{42, "hello"}

Collection Comprehensions

Arrays support comprehension syntax inside the [type].{...} form. A for expression inside the braces generates the elements. 

// Array comprehension — produce 3 random floats
let weights = [f64].{ for 0..3 do random.float(f64, -1.0, 1.0) }

// Array comprehension with an iteration variable
let squares = [i32].{ for i in 1..=5 do i * i }
// .{1, 4, 9, 16, 25}

Record Literals

Record literals use dot-brace notation with = for named fields. If a local variable shares the name of a field, the field can be omitted (shorthand). 

let Vector = record {
    x: f64,
    y: f64
}

let v = Vector.{ x = 1.0, y = 2.0 }

// Field shorthand — variable name matches field name
let x = 3.0
let y = 4.0
let w = Vector.{ x, y }  // equivalent to .{ x = x, y = y }

let x = w.x

Variant Constructors

Variant values are constructed using dot-prefix notation on the variant tag name.

let Color = variant {
    red,
    green,
    blue,
    custom: [u8, u8, u8]
}

let c1 = Color.red
let c2 = Color.custom(.{255, 128, 0})

Option and Result Literals

option(T) and result(T, E) are built-in variant types with their own constructors. Shorthand is provided for these types ?(T) and !(T, E) respectively. 

let some_val: option(i32) = .some(42)
let no_val: ?(i32) = .none

let ok_val: result(i32, string) = .ok(100)
let err_val: !(i32, string) = .err("something went wrong")

Range Literals

The .. and ..= operators produce range values. The exclusive form a..b covers a up to but not including b. The inclusive form a..=b includes both endpoints. 

let exclusive = 0..10    // 0, 1, ..., 9
let inclusive = 0..=10   // 0, 1, ..., 10

// Variable bounds
let start = 2
let end = 7
for i in start..end do
    ruka.println("${i}")     // 2 3 4 5 6
end

// Slice an array with a range
let nums: [i32] = .{10, 20, 30, 40, 50}
let slice = nums[1..4]   // .{20, 30, 40}

Unwrap Methods

Both built-in variant types provide methods for force-unwrapping their inner value. These methods panic at runtime if the value is absent or an error.

SyntaxTypeBehaviour
expr.?()option(T)Returns the inner T; panics if .none.
expr.!()result(T, E)Returns the inner T; panics if .err.
 
let name = .some("Ruka")
let s = name.?()               // "Ruka" — panics if .none

let res = !(i32, string).ok(42)
let n = res.!()                // 42 — panics if .err

// Common in chains where failure is a programmer error
let count = parse_int(input).!()
let upper = find_user(id).?().name

Range Types

A range value is produced by the .. and ..= operators. Both bounds must be the same numeric type (inferred from context). Ranges implement the iterator behaviour and can be used in for loops and as array/tuple slice indices.

start..endstart ≤ i < endExclusive range — end value not included

for i in 0..5 do
    ruka.println("${i}")     // 0 1 2 3 4
end

for i in 1..=5 do
    ruka.println("${i}")     // 1 2 3 4 5
end

// Ranges as slice indices
let nums: [i32] = .{10, 20, 30, 40, 50}
let middle = nums[1..4]   // .{20, 30, 40}
let tail = nums[2..=4]    // .{30, 40, 50}

Type Annotations

Type annotations follow a colon after the identifier. They are never required where the type can be inferred — see Type Inference for the full rules. Annotations are only written to resolve genuine ambiguity or to improve readability at module boundaries. 

let x    = 10              // inferred as int — default integer type
let f = 3.14               // inferred as float  — default float type
let name = "Ruka"          // inferred as string

let y: u8 = 10             // annotation required — non-default integer width
let g: f32 = 3.14          // annotation required — non-default float width

let scores: [f64] = .{9.5, 8.0, 7.3}   // annotation required — collection literal needs context
let pair: [i32, string] = .{1, "one"}

User-defined Types

Types are values stored in let bindings. A lowercase name (e.g. vec) makes the type public; an uppercase name (e.g. Vec) makes it private to the file (see Privacy). A type definition describes the memory layout only — behaviour (methods and static functions) is declared separately.

Records

Records are product types (similar to structs). Fields are separated by commas. A field is private when its name begins with an uppercase letter (see Privacy).

let Vector = record {
    x: f64,
    y: f64
}

let User = record {
    name: string,
    email: string,
    PasswordHash: string  // private field — uppercase first letter
}

// Inline (anonymous) record types are also valid
let origin: record { x: f64, y: f64 } = .{ x = 0.0, y = 0.0 }

Record Destructuring

A record value can be destructured into individual bindings using the .{…} syntax on the left-hand side of a binding. The binding keyword determines visibility and mutability of the resulting names. This is the same mechanism used when destructuring imported files. 

let Point = record { x: f64, y: f64 }

let p = Point.{ x = 3.0, y = 4.0 }

// Destructure into separate bindings
let { x, y } = p
// x is 3.0, y is 4.0

// Works with any record — including imported files
let { sqrt, pow } = ruka.import("math")
let { Dog, Cat }  = ruka.import("animals")

Variants (Tagged Unions)

Variants are sum types. Each tag can optionally carry a payload. A tag with no payload has the unit type. 

let Shape = variant {
    circle: f64,               // payload: radius
    rectangle: [f64, f64],     // payload: [width, height]
    point                      // no payload
}

// Variants can be nested or reference other types
let Tree = variant {
    leaf: i32,
    node: record { left: Tree, right: Tree }
}

// Inline variants inside a record field
let Cat = record {
    name: string,
    colour: variant {
        black,
        white,
        grey
    }
}

Type Inference

Ruka uses aggressive bidirectional type inference. Type annotations are only required to resolve genuine ambiguity — the compiler reports a hard error when it cannot determine a unique type. There is no implicit fallback type and no silent coercion.

Methods and static functions must be declared in the same scope or module as their type. This is what makes inference possible: the compiler always knows the complete set of types to consider when resolving an unannotated expression.

Method Receiver Type Inference

Because self is a keyword, the compiler always knows a declaration is a method. What it still needs to determine is which type self belongs to. If self.fieldName is accessed and exactly one type in scope has that field, the receiver type is resolved without an annotation. When the field exists on more than one type, an explicit annotation (self: TypeName) is required — and the compiler will tell you. 

let Circle = record { radius: f64 }
let Rect = record { width: f64, height: f64 }

// self inferred as Circle — only type in scope with `radius`
// return type inferred as f64 from the expression
let area (self) = () do 3.14159 * self.radius * self.radius

// self inferred as Rect — only type with both `width` and `height`
let area (self) = () do self.width * self.height

let Dog = record { name: string, breed: string }
let Cat = record { name: string, indoor: bool }

// Inferred as Dog — only Dog has both `name` and `breed`
let label (self) = () do "${self.name} (${self.breed})"
// Explicit annotation required — both types have a `name` field
let label (self: Cat) = () do "${self.name}"

Record Literal Inference

A record literal .{ field = value, ... } infers its type from the set of fields provided. If exactly one type in scope has that field set, no type prefix is needed. This applies to bindings, function arguments, and return expressions. 

let Circle = record { radius: f64 }
let Rect = record { width: f64, height: f64 }

// Inferred as Circle — only type with `radius`
let c = .{ radius = 5.0 }

// Inferred as Rect — only type with `width` and `height`
let r = .{ width = 4.0, height = 6.0 }

let Dog = record { name: string, breed: string }
let Cat = record { name: string, indoor: bool }

let d  = .{ name = "Rex",      breed = "Labrador" }  // Dog — Cat has no `breed`
let c2 = .{ name = "Whiskers", indoor = true }        // Cat — Dog has no `indoor`

Return Type Inference

Return types are inferred from the last expression in the function body. Explicit return type annotations are only needed when the inferred type is ambiguous, or as documentation at a public API boundary. 

// Return type inferred as f64
let circumference (self) = () do 2.0 * 3.14159 * self.radius

// Return type inferred as string — interpolation always produces string
let label (self) = () do "Circle(r=${self.radius})"

// Return type inferred as option(i32) from the .some / .none arms
let find = (values: [i32], target: i32) do
    for i in 0..values.len do
        if values[i] == target do return .some(i)
    end
    .none
end

Numeric Literal Defaults

Numeric literals have default types that apply when no context provides a more specific type.

LiteralDefault typeOverride example
Integer (42, -7, 0xFF) int let x: u8 = 42
Float (3.14, -0.5) float let x: f32 = 3.14 
let a = 42          // int
let b = 3.14        // float

// Context overrides the default
let c: u8  = 42    // u8  — annotation provides context
let d: f64 = 3.14  // f64 — annotation provides context

// Function parameter type provides context
let double = (x: i32) do x * 2
let result = double(10)  // 10 is produced as i32, not int
Ambiguity is always a compile error. The compiler will highlight the ambiguous expression and report which types are candidates. Resolve it by adding a type annotation at the point of ambiguity.

Control Flow

All control flow constructs in Ruka are expressions — they produce a value. Logical operators use keywords: and, or, not.

If / Else

do follows if and else if to separate the condition from the body. else is followed directly by its body — no do. A body whose body is a single expression on the same line is closed by the newline; a body that opens a new line is a multi-statement block and is closed by the chain's single trailing end. If an if expression is used as a value, all branches must return the same type. 

// Multiline — one `end` closes the whole chain
if x > 0 do
    ruka.println("positive")
else if x < 0 do
    ruka.println("negative")
else
    ruka.println("zero")
end

// Single-line — body is on the same line; newline closes the branch
if x > 0 do ruka.println("positive")

// Single-line with else
if x > 0 do ruka.println("positive") else ruka.println("non-positive")

// Single-line chained
if x > 0 do "positive" else if x < 0 do "negative" else "zero"

// Used as a value (do...end form)
let label = if score >= 90 do
    "A"
else if score >= 80 do
    "B"
else
    "C"
end

// Logical operators
if condition and not other_condition do
    // ...
end

if a or b do
    // ...
end

Conditional Expression (Ternary)

A right-hand-side conditional uses the same keywords rearranged: value if cond else value. It is right-associative and may be chained to express multi-way selection inline. Both branches must produce the same type. 

// Pick a value inline
let label = "positive" if x > 0 else "non-positive"

// Chained — right-associative; reads like "if/else if/else"
let grade =
    "A" if score >= 90 else
    "B" if score >= 80 else
    "C"

// Useful in single-expression function bodies
let fib = (n) do n if n <= 1 else fib(n - 1) + fib(n - 2)

Bitwise Operators

Bitwise operators work on integer types. They operate on the raw bit representation of the value. 

let a = 0b1100   // 12
let b = 0b1010   // 10

let and_result = a & b    // 0b1000 — 8
let or_result  = a | b    // 0b1110 — 14
let xor_result = a ^ b    // 0b0110 — 6

Exponentiation

The ** operator raises the left operand to the power of the right. It is right-associative, so 2 ** 3 ** 2 evaluates as 2 ** (3 ** 2). 

let squared = 4 ** 2     // 16
let cubed   = 2 ** 3     // 8

// Right-associative
let nested = 2 ** 3 ** 2   // 2 ** 9 = 512

Pattern Matching

The match expression compares a value against a series of arms. Arms are tested in order. The result of a match is the value of the matched arm.

Every arm is opened by do. A single-expression arm keeps the body on the same line as do and is closed by the newline. A multi-statement arm opens a new line after do and is closed by an explicit end. The else arm is the catch-all default.

Matching literals and conditions

match status_code with
    200 do ruka.println("OK")
    404 do ruka.println("Not Found")
    500 do
        ruka.println("Internal Server Error")
        log_error()
    end
    else ruka.println("Unknown status")
end

Matching and binding variant payloads

let Shape = variant {
    circle: f64,
    rectangle: [f64, f64],
    point
}

let area = match shape with
    circle(r) do 3.14159 * r * r
    rectangle(.{w, h}) do w * h
    point do 0.0
end

// option(T)
match maybe_name with
    some(name) do ruka.println("Hello, ${name}")
    none do ruka.println("No name provided")
end

Guard conditions

Arms may use boolean conditions involving the matched value or any in-scope variable.

match n with
    0 do "zero"
    n < 0 do "negative"
    n > 100 do "large"
    else "other"
end

Loops

While Loop

let *i = 0
while i < 10 do
    ruka.println("${i}")
    i = i + 1
end

// Single-expression body — newline closes the block
while i < 10 do i = i + 1

Destructuring Conditions

A pattern match can serve as the condition of an if or while using the pattern = expr form. The condition is true only if the pattern matches, and any bindings it introduces are in scope for the body. 

// if with destructuring — body only runs if the option is .some
let maybe: option(i32) = .some(42)
if some(x) = maybe do
    ruka.println("got ${x}")
end

// while with destructuring — loop until the value is .none
while some(x) = next_value() do
    process(x)
end

For Loop

Iterates over any value that implements the iterator behaviour. The most common iterable is a range literal. 

// Range — the common numeric loop idiom
for i in 0..5 do
    ruka.println("${i}")     // 0 1 2 3 4
end

// Single-expression body — newline closes the block
for i in 0..5 do ruka.println("${i}")

// Range without a binding variable
for 0..10 do ruka.println("tick")

// Inclusive range
for i in 1..=3 do
    ruka.println("${i}")     // 1 2 3
end

// Iterating over an array
let numbers = .{10, 20, 30}
for n in numbers do
    ruka.println("${n}")
end

// Destructuring a tuple in a for loop
let pairs: [[i32, string]] = .{ .{1, "one"}, .{2, "two"}, .{3, "three"} }

for {k, v} in pairs do
    ruka.println("${k} = ${v}")
end

Pattern-Filtered For Loop

A pattern in the for loop variable filters the sequence: only elements that match the pattern are visited. Non-matching elements are skipped silently. 

let values: [option(i32)] = .{.some(1), .none, .some(3), .none, .some(5)}

// Only visits elements where the option is .some; x is bound to the inner value
for some(x) in values do
    ruka.println("${x}")    // 1, 3, 5
end

Break

break immediately exits the innermost enclosing while or for loop. 

// Exit a while loop early
let *i = 0
while true do
    if i >= 5 do break
    ruka.println("${i}")
    i = i + 1
end

// Stop searching once a match is found
let *found = false
for item in items do
    if item == target do
        found = true
        break
    end
end

Continue

continue skips the remainder of the current loop iteration and moves immediately to the next one. 

// Print only odd numbers
for i in 0..10 do
    if i % 2 == 0 do continue
    ruka.println("${i}")     // 1 3 5 7 9
end

// Skip invalid entries while processing
for entry in entries do
    if not entry.is_valid() do continue end
    process(entry)
end

Defer

A defer statement schedules an expression to run at the end of the enclosing do…end block, regardless of how control exits that block — whether by reaching the end, an early return, or a break. This makes defer the idiomatic way to pair cleanup with acquisition.

When multiple defer statements appear in the same block they execute in LIFO order — the last one declared runs first. 

// Guaranteed cleanup: close() always runs when the block exits
let read_file = (path: string) do
    let file = open(path)
    defer file.close()

    // ... work with file
end   // file.close() is called here

// Multiple defers execute in reverse declaration order
let example = () do
    defer ruka.println("third")    // runs last
    defer ruka.println("second")
    defer ruka.println("first")    // runs first
    // Output on exit: first, second, third
end

// Defer works at any scope depth — it binds to the nearest enclosing block
let process = (items: [i32]) do
    let resource = acquire()
    defer resource.release()       // released when process() returns

    for item in items do
        let handle = item_handle(item)
        defer handle.close()       // closed at the end of each loop iteration
        handle.transform()
    end
end
defer captures variables by reference at the point of declaration. The deferred expression sees the variable values as they are when the block exits, not when defer was written.

Functions

Functions are anonymous values stored in bindings. There is no special fn keyword — a function is written as a parameter list followed by a body block or single expression.

The last expression in a block is returned. An early return can be written with the return keyword. return always requires a payload expression — a function whose return type is () writes return () to exit early, where () is the unit literal.

Multiline function (do…end block)

let add = (a: i32, b: i32) do
    a + b
end

let greet = (name: string) do
    let msg = "Hello, ${name}!"
    ruka.println(msg)
    // last expression is (), the unit value
end

Single-expression function

When the body is a single expression on the same line as do, the newline closes the function — no end is required. For bodies that span multiple lines, the do…end block form is used. 

let square = (x: i32) do x * x

let abs = (x: i32) do if x < 0 do -x else x

// Multi-line body — closed by `end`
let max = (a: i32, b: i32) do
    if a > b do a else b
end

Return type annotation

A -> after the parameter list specifies the return type. This applies to both single-expression and do…end bodies. 

// Inline — return type after the parameter list
let square = (x: i32) -> i32 do x * x

// Works identically with multi-line bodies
let clamp = (x: i32, lo: i32, hi: i32) -> i32 do
    if x < lo do lo else if x > hi do hi else x
end

// Binding-side annotation — function type written as parameter list then return type
let square2: (i32) -> i32 = (x) do x * x

Automatic Return Wrapping

When a function's return type is annotated as option(T) or result(T, E), the compiler automatically wraps returned values rather than requiring explicit .some(...) or .ok(...) at every return site. The wrapping is inferred from the type of the returned expression:

  • option(T): a T value is wrapped as .some(T); a () expression becomes .none.
  • result(T, E): a T value is wrapped as .ok(T); an E value is wrapped as .err(E).
// Returns option(uint) — wrapping is automatic
let find_index = (arr: [i32], target: i32) -> option(uint) do
    for i in 0..arr.len do
        if arr[i] == target do return i              // i (uint) → .some(i)
    end
    // () at end of block → .none
end

// Returns result(f64, string) — wrapping is automatic
let safe_div = (a: f64, b: f64) -> result(f64, string) do
    if b == 0.0 do return "division by zero"        // string → .err(...)
    a / b                                           // f64   → .ok(...)
end

Entry point

The program entry point is a public constant named main.

let main = () do
    ruka.println("Hello, world!")
end

Static Functions

A static is any value associated with a type using the receiver syntax — a type name in parentheses between the binding name and the = sign. Any value can be attached as a static, not just functions: constants, default instances, and lookup tables are all valid. Using a type name in the receiver position — rather than the reserved keyword self — is what distinguishes a static from a method. 

let Vector = record {
    x: f64,
    y: f64
}

// Public static function — called as Vector.new(...)
let new (Vector) = (x, y) do
    .{ x, y }
end

// Private static function
let zero (Vector) = () do .{ x = 0.0, y = 0.0 }

let main = () do
    let v = Vector.new(1.0, 2.0)
    let origin = Vector.zero()
end
Fields vs. statics: A type's fields are runtime instance attributes — they exist on each value of that type and are accessed via value.field. Statics are compile-time shared attributes associated with the type itself, not with any particular instance, and are accessed via TypeName.static.

Non-Function Statics

Any compile-time value — a constant, a default instance, a type — can be attached as a static. Access uses the same TypeName.name dot notation. 

let Color = record { r: u8, g: u8, b: u8 }

// Constant values attached as statics
let RED (Color) = Color.{ r = 255, g = 0,   b = 0   }
let GREEN (Color) = Color.{ r = 0,   g = 255, b = 0   }
let BLUE (Color) = Color.{ r = 0,   g = 0,   b = 255 }

let main = () do
    let c = Color.RED
    ruka.println("r=${c.r}")   // r=255
end

Static Functions on Variants

Static functions can be attached to variant types using the same receiver syntax. This is useful for named constructors and factory helpers. 

let Direction = variant {
    north,
    south,
    east,
    west
}

let from_str (Direction) = (s) do
    match s with
        "north" do .some(Direction.north)
        "south" do .some(Direction.south)
        "east"  do .some(Direction.east)
        "west"  do .some(Direction.west)
        else    .none
    end
end

let main = () do
    let d = Direction.from_str("east")
end
Any type can carry methods and statics. The receiver syntax works uniformly across every type in the language: user-defined records and variants, built-in generics (option(T), result(T, E), arrays, ranges), and primitives (i32, f64, bool, string, …) all accept methods with self receivers and statics with type-name receivers. A primitive is simply a type whose layout is fixed by the compiler; it has no privileged or restricted status with respect to behaviour.

Constructors & Destructors

Ruka has no special constructor or destructor syntax. Instead, idiomatic constructors are plain static functions named new attached to the type, and idiomatic destructors are methods named deinit that take ownership of the receiver using the & (move) mode prefix.

Idiomatic Constructor

A constructor is a static function named new. It allocates and initialises a value of the associated type, then returns it. Any additional setup logic lives here. 

let Buffer = record {
    data: [u8],
    len:  uint,
}

// Public constructor — called as Buffer.new(capacity)
let new (Buffer) = (cap: uint) do
    Buffer.{
        data = [u8; cap].{},
        len  = 0,
    }
end

let main = () do
    let buf = Buffer.new(64)
end

Idiomatic Teardown

A teardown method is named deinit and takes the receiver by move (&self). The & mode transfers ownership into the function — the caller cannot use the value after calling deinit, and the compiler will emit an error on any subsequent use of the moved binding.

deinit is the right place for any finalisation work: flushing a buffer, closing a file handle, returning memory to an allocator, or signalling that an object has completed its work. It is not limited to deallocation. 

// deinit — ownership of self is moved in; self cannot be used after this call
let deinit (&self) = () do
    // flush pending writes, close handle, signal completion, etc.
    ruka.println("Buffer flushed and freed")
end

let main = () do
    let buf = Buffer.new(64)
    // ... use buf ...
    buf.deinit()
    // buf.len   // compile error: buf was moved by deinit
end
Move ownership and the GC: Because deinit takes &self, ownership is transferred into the call. The GC treats the binding as dead the moment deinit returns, allowing it to reclaim the value immediately rather than waiting for the enclosing scope to end. This makes deinit useful even for fully GC-managed types — any time you want deterministic cleanup or want to shorten a value's lifetime explicitly.

Methods

A method is declared like a static function, but uses the reserved keyword self in the receiver position instead of a type name. self refers to the instance the method is called on and is always the first implicit argument. Any mode prefix (*, &, etc.) may precede self.

This is the syntactic distinction between a method and a static function: if the receiver is self it is a method; if it is a type name it is a static function. Because self is a keyword, the difference is unambiguous regardless of naming conventions. 

let Vector = record {
    x: f64,
    y: f64
}

// Public method — self is the receiver instance; its type is inferred as Vector
let add (self) = (other) do
    .{ x = self.x + other.x, y = self.y + other.y }
end

let length (self) = () do
    ruka.sqrt(self.x * self.x + self.y * self.y)
end

// Private method
let negate (self) = () do .{ x = -self.x, y = -self.y }

let main = () do
    let a = Vector.new(1.0, 0.0)
    let b = Vector.new(0.0, 1.0)
    let c = a.add(b)
    ruka.println("Length: ${c.length()}")
end

Mutable Methods

To mutate the receiver inside a method, apply the * prefix to self. The caller must hold a mutable binding. 

let Counter = record { value: i32 }

let new (Counter) = () do .{ value = 0 }

// *self — the receiver is a mutable reference
let increment (*self) = () do
    self.value = self.value + 1
end

let main = () do
    let *c = Counter.new()
    c.increment()         // value is now 1
    c.increment()         // value is now 2
    ruka.println("${c.value}")
end

Methods on Variants

Methods can also be attached to variant types using the same self receiver syntax. Pattern matching on self inside the body handles each tag. 

let Direction = variant {
    north,
    south,
    east,
    west
}

let opposite (self) = () do
    match self with
        north do Direction.south
        south do Direction.north
        east  do Direction.west
        west  do Direction.east
    end
end

let is_vertical (self) = () do
    self == Direction.north or self == Direction.south
end

let main = () do
    let d = Direction.north
    ruka.println("${d.opposite().is_vertical()}")  // true
end

Extending Types Outside Their Declaring Scope

Methods and statics can be attached to a type from any scope, not only the one where the type is declared. This applies uniformly to user types, built-in generics, and primitives. An extension never mutates the original type; instead, the extending scope gets a new type of the same name that shadows the original for the duration of that scope. The extended type has every member of the original plus the newly declared ones. Code in other scopes continues to see the unextended type.

This lets a module tailor a type to satisfy a behaviour — or add convenience helpers — without affecting any other user of that type. 

// Add a method to the primitive `i32` in this file only.
// Elsewhere in the program `i32` is unchanged.
let squared (self: i32) = () -> i32 do self * self

let main = () do
    let n: i32 = 7
    ruka.println("${n.squared()}")   // 49
end
// Extend a built-in generic locally to satisfy a behaviour.
let printable = behaviour { print_self (self) = () -> () }

let print_self (self: option(i32)) = () do
    match self with
        .some(n) do ruka.println("${n}")
        .none    do ruka.println("(none)")
    end
end

let show = (x: printable) do x.print_self()

let main = () do
    show(.some(42))   // ok — within this scope option(i32) satisfies `printable`
    show(.none)
end
Shadowing, not monkey-patching. An extension is lexically scoped: it is visible only where the extending bindings are in scope (the file they are declared in, or any scope that imports them and brings the names in). Two files may independently extend the same type in incompatible ways without conflict; each sees its own shadowed version. The underlying in-memory representation is unchanged, so extended and unextended values are interchangeable — the extension only adds dispatch entries resolvable from the extending scope.

Parameter Modes

By default, all arguments are passed as immutable references managed by the garbage collector. A mode prefix placed directly before the parameter identifier changes how the compiler treats the lifetime and ownership of that value. Modes apply to any parameter, including method receivers. Some are shared with binding modes.

PrefixSyntaxBehaviour
* (*x: T) Mutable reference. Changes to x are visible to the caller.
& (&x: T) Ownership is transferred into this function. The caller cannot use the value after the call.
$ ($x: T) Stack-allocated pass-by-copy.
@ (@x: T) or (x: type) Compile-time constant. The value must be known at compile time. Inferred automatically when the parameter type is type. 
// * — in-place mutation
let fill (*self) = (value: u8) do
    // modifies caller's buffer
end

// & — consume / transfer ownership
let close = (&handle: FileHandle) do
    // handle is invalid for the caller after this call
end

// $ — stack allocated pass-by-copy
let hash = ($data: [u8]) -> u64 do
    // data is on the stack
end

// @ — compile-time value (explicit annotation)
let repeat = (@times: uint, msg: string) do
    let *i = 0
    while i < times do
        ruka.println(msg)
        i = i + 1
    end
end
Note: The @ compile-time prefix is covered in detail in the Compile-Time Evaluation section, including automatic inference from type-annotated parameters and storing compile-time results.

Named Parameters

Prepending ~ to a parameter name makes it a named parameter. Named parameters must be passed by label at the call site using ~label=value syntax and may appear in any order. Positional and named parameters can be freely mixed.

Declaring Named Parameters

// Both parameters are named — order at call site doesn't matter
let greet = (~name: string, ~greeting: string) do
    "${greeting}, ${name}!"
end

// Mix of positional and named
let clamp = (x: int, ~lo: int, ~hi: int) do
    if x < lo do
    	lo
     else if x > hi do
     	hi
    else
    	x
    end
end

Calling with Named Parameters

greet(~name="Ruka", ~greeting="Hello")   // "Hello, Ruka!"
greet(~greeting="Hi", ~name="World")     // "Hi, World!" — order doesn't matter

let v = clamp(42, ~lo=0, ~hi=10)         // 10 — positional first, named in any order

Label Shorthand

If a local variable shares the label name, the =value part can be omitted. 

let name = "Ruka"
let greeting = "Hello"

greet(~name, ~greeting)   // shorthand for greet(~name=name, ~greeting=greeting)

Optional Named Parameters

A named parameter whose type is option(T) is optional at the call site. When the argument is omitted, the parameter receives .none automatically. When a value is supplied, it is automatically wrapped in .some(...) — no explicit wrapping is needed. 

let greet = (~name: string, ~title: option(string)) do
    "${t} ${name}" if .some(t) = title else name
end

greet(~name="Ruka")                    // "Ruka"          — title is .none
greet(~name="Ruka", ~title="Dr.")      // "Dr. Ruka"      — title is .some("Dr.")
greet(~name="Ruka", ~title=.none)      // explicit .none also valid

Trailing Block Syntax

A named parameter can be passed after the closing parenthesis as a trailing argument. This is most useful for higher-order functions that accept a closure — the named argument reads as a block visually attached to the call. Only one trailing argument is permitted per call. 

let apply_twice = (x: int, ~f: (int) -> int) do f(f(x))

// Named param inside the argument list — standard form
let r1 = apply_twice(3, ~f=(x) do x * 2)    // 12

// Trailing form — same result, reads like an attached block
let r2 = apply_twice(3) ~f=(x) do x * 2     // 12

// Multi-line trailing block
let r3 = apply_twice(3) ~f=(x) do
    let y = x * 2
    y + 1
end                                           // 14

Higher-order functions

The trailing syntax is especially readable for collection operations. Named closure parameters are placed last in the definition so callers can trail them. 

// `t: type` and `u: type` infer eva automatically
let map = (t: type, u: type, arr: [t], ~f: (t) -> u) -> [u] do ... end
let filter = (t: type, arr: [t], ~pred: (t) -> bool) -> [t] do ... end
let reduce = (t: type, u: type, arr: [t], init: u, ~f: (u, t) -> u) -> u do ... end

let nums = .{1, 2, 3, 4, 5}

let doubled = map(nums) ~f=(x) do x * 2
let evens = filter(nums) ~pred=(x) do x % 2 == 0
let total = reduce(nums, 0) ~f=(acc, x) do acc + x   // 15

// Multi-line body — the trailing block scales naturally
let results = map(nums) ~f=(x) do
    let squared = x * x
    squared - 1
end
Note: Named parameters are always identified by label, never by position. A named parameter cannot be passed without its ~label= prefix.

Behaviours

A behaviour defines a set of method signatures that a type must satisfy. Behaviours describe behaviour only — they carry no fields or state. Any type that implements all of a behaviour's methods automatically satisfies it; there is no explicit implements declaration.

Inside the braces, each line declares one method signature: the method name, its receiver (with optional mode), then the remaining parameter list and return type.

Declaring a Behaviour

// A type that can be printed
let Printable = behaviour {
    print(self): () -> ()
}

// A type that supports a binary add operation
let Adder = behaviour {
    add(mut self): (b: uint) -> ()
}

// Multiple methods
let Shape = behaviour {
    area(self): () -> f64,
    perimeter(self): () -> f64
}

Satisfying a Behaviour

A type satisfies a behaviour implicitly. As long as the required methods are declared in the same module as the type, the compiler considers the type to implement the behaviour. Receiver and return types are typically inferred — annotations are only needed when the field set is ambiguous. 

let Circle = record { radius: f64 }

// self inferred as Circle (only type in scope with `radius`);
// return types inferred from the expressions
let area (self) = () do 3.14159 * self.radius * self.radius
let perimeter (self) = () do 2.0 * 3.14159 * self.radius

// Circle now satisfies the Shape behaviour — no declaration needed

Monomorphisation via Behaviour Types

When a parameter's type is a behaviour, a version of the function will be compiled for that specific argument's type. Any value whose type satisfies the behaviour can be passed. Behaviour types are never inferred — they must always be written explicitly as a type annotation. 

// Behaviour type annotation on a parameter — dynamic dispatch is automatic
let print_shape_info = (s: Shape) do
    ruka.println("area: ${s.area()}")
    ruka.println("perimeter: ${s.perimeter()}")
end

let Rect = record { w: f64, h: f64 }

let area (self) = () do self.w * self.h
let perimeter (self) = () do 2.0 * (self.w + self.h)

let main = () do
    let c = Circle.{ radius = 5.0 }
    let r = Rect.{ w = 4.0, h = 6.0 }

    print_shape_info(c)   // Circle satisfies Shape
    print_shape_info(r)   // Rect satisfies Shape
end
Note: behaviours specify only methods. Fields cannot appear in a behaviour definition. To share field access across types, expose it through methods.
Behaviour types may only appear in parameter positions. Using a behaviour as a return type, field type, or binding type is a compile error. Behaviours describe opaque behaviour — they cannot be stored, only passed into functions.

Builtin Behaviours

The ruka module exposes a small set of standard behaviours that the compiler and runtime recognise specially. Satisfying them unlocks language-level features: string interpolation, for loops, and custom allocator support. Like all behaviours, they are satisfied implicitly — no declaration is needed.

ruka.printable

Any type that satisfies ruka.printable can be interpolated inside a ${…} string expression and passed to ruka.print / ruka.println. The compiler calls format automatically in those positions. 

let printable = behaviour {
    format(self): () -> string
}

// Example — make a Point printable
let Point = record { x: f64, y: f64 }

let format (self) = () do "(${self.x}, ${self.y})"

let main = () do
    let p = Point.{ x = 1.0, y = 2.0 }
    ruka.println("point: ${p}")   // point: (1.0, 2.0)
end

ruka.iterable

Any type that satisfies ruka.iterable can be used in a for loop. The compiler calls next on each iteration; when it returns .none the loop ends. 

let iterable = behaviour {
    next(*self): () -> ?(T)
}

// Example — a simple counter iterator
let Counter = record { current: uint, limit: uint }

let next (*self) = () do
    if self.current >= self.limit do .none
    else
        let val = self.current
        self.current = self.current + 1
        .some(val)
    end
end

let main = () do
    let *c = Counter.{ current = 0, limit = 3 }
    for n in c do
        ruka.println("${n}")   // 0, 1, 2
    end
end

ruka.allocator

Any type that satisfies ruka.allocator can be passed to functions or data structures that need explicit heap control. This allows callers to supply arena allocators, pool allocators, or any other strategy without coupling the implementation to a specific allocator. 

let allocator = behaviour {
    alloc  (*self): (@size: uint) -> [u8],
    free   (*self): (&bytes: [u8]) -> ()
}

// A function that accepts any allocator
let make_buffer = (alloc: ruka.allocator, cap: uint) do
    Buffer.{ data = alloc.alloc(cap), len = 0 }
end
ruka.allocator and the GC: The default allocator used by the runtime is the GC. Passing a custom ruka.allocator opts a specific allocation out of GC management — the returned buffer is an ordinary reference, but the GC does not track it, so the caller is responsible for calling free (typically via deinit).
GC exclusion in manual-memory programs: If a program never uses GC-allocated values — every heap allocation goes through ruka.heap_alloc or a custom ruka.allocator — the compiler statically proves the GC has no roots to scan and excludes the GC runtime from the final binary entirely. This makes Ruka practical for embedded targets and low-latency code where a GC pause is unacceptable.

Pipelines

The pipeline operator |> passes the left-hand value as the first argument to the right-hand function. It allows chaining operations in the order they are applied, which is easier to read than deeply nested calls. 

// Without pipeline
ruka.println(trim(toUpperCase(getName(user))))

// With pipeline — reads top to bottom, left to right
getName(user)
    |> toUpperCase()
    |> trim()
    |> ruka.println()

Modules

A file is a record. There is no separate module concept — every Ruka source file implicitly defines a compile-time record value whose fields are the file's top-level let bindings. Bindings whose names begin with a lowercase letter are public fields; those whose names begin with an uppercase letter are private and not accessible to importers (see Privacy).

Because files are records, all of the language's record machinery applies directly: field access retrieves exports, and destructuring pulls named fields into scope. Name case determines privacy consistently whether on a record field or a file-scope binding.

A file record is an empty record — it has no instance fields, only statics. It cannot be instantiated with .{}. Every declaration in the file is stored as a static field accessible via the record value returned by ruka.import. 

// stack.ruka — a file is a record; lowercase-named top-level bindings are public fields

let Stack = record {     // uppercase name — private to this file
    Data: [i32],         // uppercase field — private
    Len:  uint
}

let new (Stack) = () do .{ Data = .{}, Len = 0 }    // public field

let push (*self) = (value: i32) do                  // public field
    // append value to self.Data
    self.Len = self.Len + 1
end

let pop (*self) = () -> option(i32) do              // public field
    if self.Len == 0 do
        .none
    else
        self.Len = self.Len - 1
        .some(self.Data[self.Len])
    end
end

Imports

ruka.import evaluates at compile time and returns the file's record value. The result is always compile-time known — compile-time evaluation is implied. Accessing an export is ordinary field access; destructuring an import is ordinary record destructuring, using the same {…} syntax. 

// Field access on the imported record
let math = ruka.import("math")

let main = () do
    ruka.println("${math.sqrt(2.0)}")
end

// Record destructuring — pull specific fields into scope
let { sqrt, pow } = ruka.import("math")
let { Dog, Cat }  = ruka.import("animals")

// Imported records can be passed to eva parameters like any compile-time value
let describe = (t: type) do ruka.println("${t}")
let animals = ruka.import("animals")

let main = () do
    describe(animals.Dog)   // passes the Dog type to a compile-time parameter
end

Test Bindings

The test keyword declares a test function. Test bindings follow the same syntax as any other binding, but with two important differences: the value must be a zero-argument function, and the binding is compiled only in debug and test builds. In optimised builds test declarations are elided entirely — they contribute no code and no overhead to production binaries.

Tests are never exported. They can access everything in their module's scope, including uppercase-named (private) bindings, which makes it straightforward to test internal implementation details without exposing them. 

let add = (a: i32, b: i32) do a + b

test addition = () do
    ruka.assert_eq(add(1, 2), 3)
    ruka.assert_eq(add(-1, 1), 0)
    ruka.assert_eq(add(0, 0), 0)
end

test addition_is_commutative = () do
    ruka.assert_eq(add(3, 4), add(4, 3))
end

Because tests live in the same scope as the module, they have access to uppercase-named (private) definitions directly — no special test-visibility mechanism is needed. 

// Private implementation detail — not exported
let clamp_internal = (x: i32, lo: i32, hi: i32) do if x < lo do lo else if x > hi do hi else x

test clamp_lower_bound = () do
    ruka.assert_eq(clamp_internal(3, 5, 10), 5)
end

test clamp_upper_bound = () do
    ruka.assert_eq(clamp_internal(15, 5, 10), 10)
end

test clamp_in_range = () do
    ruka.assert_eq(clamp_internal(7, 5, 10), 7)
end
Compilation modes. test bindings are included in two build modes: debug (full code coverage analysis, all assertions active) and test (runs the test suite and reports pass/fail). They are stripped in the release build. This means ruka.expect_eq and other assertion calls inside a test binding never appear in a production binary.

Compile-Time Evaluation

Ruka includes a compile-time interpreter. Functions with eva parameters are evaluated at compile time rather than at runtime. This enables generic programming through static dispatch and the generation of complex types that cannot be expressed as simple literals. Types, functions, and modules are first-class values in compile-time contexts.

Automatic Compile-Time Inference

When a parameter's type annotation is type, the compile-time (@) mode is inferred automatically — no explicit prefix is needed. Writing (t: type) is exactly equivalent to (@t: type). When a parameter is used as a type annotation for another parameter, both the @ prefix and the type annotation are inferred. 

// Explicit — valid, but redundant
let min = (@t: type, a: t, b: t) do if a < b do a else b

// Inferred — preferred; `@` is implied by the `type` annotation
let min = (t: type, a: t, b: t) do if a < b do a else b

// Both `@` and `: type` inferred — `t` is used to annotate `a` and `b`
let min = (t, a: t, b: t) do if a < b do a else b

// Compile-time type constructor — same rules apply
let Pair = (a, b) do
    record { first: a, second: b }
end

Generics via Compile-Time Parameters

A function with a type-annotated parameter is instantiated separately for each unique set of compile-time arguments, similar to monomorphisation in Rust or comptime in Zig. 

let min = (t, a: t, b: t) do if a < b do a else b

let main = () do
    let x = min(i32, 3, 7)     // compiler emits an i32 version
    let y = min(f64, 1.5, 2.0) // compiler emits an f64 version
end

Generating Types at Compile Time

Compile-time functions can return types, enabling parameterised type definitions. Non-type parameters that must be compile-time constants still require an explicit @ prefix. 

// Returns a record type — `cap` requires explicit @ because it is not type-annotated as `type`
let FixedArray = (t, @cap: uint) do
    record {
        data: [t],
        len:  uint
    }
end

// Instantiate at compile time — these are concrete types
let IntBuf   = FixedArray(i32, 64)
let FloatBuf = FixedArray(f64, 16)

let main = () do
    let buf = IntBuf.{ data = .{}, len = 0 }
end

Storing Compile-Time Results

If a function has any @-prefixed parameters, it runs at compile time and its result is automatically a compile-time constant. At file scope the binding is implicitly compile-time, so no mode prefix is required. Inside an inner scope the binding must be declared with @, since the default there is runtime. Results can also be used inline without a binding. 

let Pair = (a, b) do
    record { first: a, second: b }
end

// File scope — compile-time is inferred
let IntStrPair = Pair(i32, string)

let main = () do
    // Inner scope — @ is required to store the compile-time result as a constant
    let @IntStrPair = Pair(i32, string)

    // Used inline without a binding — also valid
    let p = Pair(i32, string).{ first = 1, second = "one" }
end

Forcing Runtime Functions at Compile Time

Inside an inner scope, the @ mode on a binding forces any function — including ordinary runtime functions — to execute at compile time. All arguments must be compile-time known. This is distinct from calling a function that already has @-prefixed parameters, which runs at compile time automatically. At file scope @ is redundant: top-level bindings are already evaluated at compile time. 

let PI = 3.14159265358979

// File scope — compile-time is implicit; ruka.sqrt runs at compile time here
let SQRT_2: f64 = ruka.sqrt(2.0)
let INV_PI: f64 = 1.0 / PI

// Chained use is still compile-time
let DOUBLE_SQRT_2 = SQRT_2 * 2.0

// Useful for any expensive or non-trivial constant
let LOOKUP = generate_lookup_table(256)
let KEY_HASH = compute_hash("app-secret")

let run = () do
    // Inside a function body, @ is required to force compile-time evaluation
    let @table = generate_lookup_table(256)
    // ...
end

Compile-Time Reflection

ruka.type_of returns the type of any expression as a compile-time value, which can be passed to @-prefixed parameters or compared at compile time. 

let assertSameType = (a: type, b: type) do
    if not (a == b) do
        ruka.compile_error("type mismatch: expected ${a}, got ${b}")
    end
end

let main = () do
    let x: i32 = 10
    let y: i32 = 20
    assertSameType(ruka.type_of(x), ruka.type_of(y))  // ok — both i32
end
Note: Only values that are provably known at compile time may be passed to @-prefixed parameters. Passing a runtime value to an @-prefixed parameter is a compile error. Compile-time known values include: literals, top-level let bindings, and results of other compile-time evaluations.

Compile-Time Type Introspection

At compile time, a type's structure can be inspected — its fields, methods, and static members are all readable as ordinary compile-time data. Combined with ruka.record_of, this enables derive-macro-style generation: a compile-time function analyses an existing type and produces a new type mechanically derived from it. 

// Given any record type, produce a version where every field is option(T).
let Partial = (t: type) do
    let fs = ruka.fields_of(t)
    let opt_fs = fs |> map((f) do .{ name = f.name, type = option(f.type) })
    ruka.record_of(opt_fs)
end

let User        = record { id: i32, name: string, email: string }
let PartialUser = Partial(User)
// PartialUser is equivalent to:
//   record { id: option(i32), name: option(string), email: option(string) }

Because the result of any @-parameterised function is a compile-time value, the synthesised type can be bound with let (or let) at file scope and used exactly like a type written by hand — instantiated with .{…}, checked by the type system, and exported from a module.

ruka.methods_of and ruka.statics_of complete the introspection surface. They enable generation patterns that also carry behaviour — for example, a function that inspects a record's fields and emits a derived record together with a matching set of accessor or serialisation methods.

Introspection is compile-time only. ruka.fields_of, ruka.methods_of, ruka.statics_of, and ruka.record_of all take @-prefixed type arguments and run during compilation. There is no runtime reflection — everything is erased before code generation, so introspection has zero runtime cost.
